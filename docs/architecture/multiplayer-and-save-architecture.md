# RadishCatalyst / 异星催化 联机模式与存档架构前置设计

## 1. 文档目的

这份文档用于在项目正式开始开发前，提前锁定联机模式、存档归属、同步方式和架构预留原则。

项目正式名称：

- 英文名：`RadishCatalyst`
- 中文名：`异星催化`

目标不是现在立刻实现完整联机，而是：

- 提前决定联机的最终方向
- 避免前期按纯单机写死，后期大改甚至重构
- 为客户端、数据结构、系统边界和后端接入方式设定约束

这份文档应被视为后续功能设计和代码结构设计的前置原则文档。

## 2. 最终联机方向

本项目在首版和首个存档 + 联机版本中不按 MMO 大世界路线设计。

远期可以保留大型多人在线的演进可能，但它不是当前阶段、首版或首个联机版本的设计前提。

本项目的联机目标正式定义为：

**类似《我的世界》的多形态联机模型：支持离线单机、本地存档、玩家开房协作、专用服务器长期存档，以及后续可选的官方账号与官方云存档。**

当前建议的协作人数范围：

- 首版设计基准：`2 ~ 5 人`
- 长期可扩展上限：`2 ~ 10 人`

这个方向的核心价值是：

- 单机先成立
- 联机后扩展体验
- 不把项目一开始拖入 MMO 级复杂度
- 同时保留长期商业化和官方服务扩展空间

人数口径的含义：

- `2 ~ 5 人` 是玩法、UI、平衡和同步复杂度的默认设计目标。
- `2 ~ 10 人` 是专用服务器或高性能主机房间的远期扩展上限，不作为首版承诺。
- 不应为了支持 10 人而提前设计团本、职业铁三角或复杂团队调度。

## 3. 联机形态分层

项目应按四层联机形态设计。

### 3.1 第一层：离线单机

特点：

- 本地运行
- 本地存档
- 不依赖网络
- 所有核心玩法完整可玩

要求：

- 没有网络时游戏也必须成立
- 所有核心玩法不能依赖在线服务
- 单机模式是整个项目的最小闭环基础

### 3.2 第二层：玩家开房 / 局域网 / 好友联机

特点：

- 一名玩家作为 Host
- 其他玩家加入房间
- 适合 2~5 人协作
- 以房主存档作为世界状态来源
- 远期可以在性能和同步验证后扩展到 2~10 人

适用玩法：

- 协作探索
- 协作战斗
- 小队远征
- 基地共同建设
- Boss 战

### 3.3 第三层：专用服务器

特点：

- 玩家自建或租用服务器
- 服务器长期保存世界状态
- 多人可以在固定世界中持续协作

适用目标：

- 固定合作档
- 稳定队伍长期推进
- 避免房主不上线时无法进入存档

### 3.4 第四层：官方服务

特点：

- 官方账号体系
- 官方云存档
- 官方服务器列表
- 好友与邀请系统
- 官方服务器或官方中继服务

说明：

- 这是中后期扩展目标
- 不是首版开发前提
- 不能成为前期核心玩法成立的依赖条件

## 4. 为什么当前不按 MMO 大世界设计

不建议在首版或首个存档 + 联机版本中把这个项目定义成 MMORPG，原因如下：

- 自动化基地状态同步本身就很复杂
- ARPG 战斗已经有较高实时同步要求
- 大世界多人在线会放大网络、存档、权限、反作弊和运维压力
- 个人开发者很难在首版同时做好基地模拟、实时战斗和 MMO 级在线系统

因此本项目的联机目标更适合定义为：

**当前阶段先做房间制 / 专服制的协作联机 ARPG，并为更远期的大型多人在线保留架构口子。**

## 5. 推荐的联机核心玩法边界

联机应该优先增强这些体验：

- 组队外出探索
- 协作清理高危区域
- 一起打精英怪和 Boss
- 分工采集与运回资源
- 在同一基地内分工建设和维护
- 协作完成高阶工业链和区域推进

联机不应该优先做这些内容：

- 大规模 PvP
- 开放式自由交易经济
- 大型公会战
- 全服共享世界资源刷新
- 长期数百人同屏公共地图

## 6. 推荐的世界与房间模型

最稳妥的模型是：

- 一个世界对应一个存档
- 一个存档可以离线单机游玩
- 该存档也可以被房主作为联机房间打开
- 后续同一个存档也可以迁移到专用服务器

这意味着世界的本质应该是：

- 同一份规则
- 同一套状态结构
- 不同运行方式

也就是：

- 单机运行时：本地模拟 + 本地存档
- Host 联机时：本地主机模拟 + 远程玩家加入
- 专服联机时：服务器模拟 + 客户端连接

这是整个项目未来避免重构的关键。

## 7. 存档归属模型

建议明确区分三类存档或档案：

### 7.1 世界存档

用于保存：

- 地图进度
- 建筑布局
- 设备状态
- 管网状态
- 储罐内容
- 电网状态
- 区域解锁
- 主线推进
- 世界任务状态
- 基地库存

它代表的是“这个世界本身”。

### 7.2 玩家档案

用于保存：

- 账号显示名或头像设置
- 输入设置
- 辅助功能设置
- 已知服务器、好友和账号信息
- 成就或全局解锁

它代表的是“这个玩家自己”。

玩家档案不应直接等同于某一个角色。一个玩家可以拥有多个角色。

### 7.3 角色档案

用于保存：

- 角色 ID
- 角色名和外观
- 初始背景或角色原型
- 初始能力
- 初始天赋
- 等级
- 技能
- 装备
- 快捷栏
- 个人背包
- 个人任务进度
- 与某个世界的绑定关系

它代表的是“这个玩家当前使用的具体角色”。

### 7.4 单机模式下的关系

单机时，世界存档和玩家档案可以一起保存在本地。

同一个玩家可以在不同本地世界中选择不同角色。不同角色可以拥有不同初始能力和天赋，用于支持不同 Build、不同开局体验和不同协作分工。

### 7.5 开房模式下的关系

开房时：

- 世界存档归房主
- 其他玩家带入自己的角色档案，选择一个角色加入，或使用房间规则指定的临时角色

建议优先考虑三种模式：

1. 角色绑定世界  
进入某个世界时，角色成长跟随该世界推进

2. 角色弱绑定世界
角色可以由玩家选择进入多个世界，但等级、装备、背包或关键进度根据世界规则限制

3. 临时房间角色
加入某个世界时按房间规则生成临时角色，离开后只保存有限个人进度或不保存

从开发复杂度和平衡看，首版建议优先：

**角色与世界弱绑定，但跨世界资源、装备、配方和任务推进携带要严格限制。**

推荐规则：

- 同一个玩家可以创建多个角色。
- 不同存档可以选择不同角色。
- 同一角色进入不同世界时，可以保留外观、基础天赋和部分个人成长。
- 资源、装备、基地解锁、区域进度和主线推进默认跟随世界存档。
- 房主或服务器应能决定是否允许外部角色加入。
- 如果允许外部角色加入，应有等级、装备、背包或材料携带限制，避免破坏世界进度。

### 7.6 初始能力与天赋

不同角色可以有不同初始能力和天赋，但它们应服务协作和 Build 倾向，而不是变成不可逆职业锁死。

推荐初始倾向：

- 工程倾向：建造、修复、部署装置或设备排障更强。
- 外勤倾向：移动、采集、扫描或背包效率更强。
- 战斗倾向：武器操控、输出、控场或弱点打击更强。
- 生存倾向：环境抗性、治疗、撤离或补给效率更强。

设计边界：

- 初始天赋不能替代基地生产和装备成长。
- 不同角色应能通过装备、模块和技能形成混合 Build。
- 首版不做复杂职业树，优先做少量清晰初始差异。
- 联机协作可以鼓励分工，但单机角色必须也能完整通关首版闭环。

### 7.7 本地存档阶段策略

当前原型已经具备默认槽位单文件存档、三槽位原型 UI 入口、最近 3 份 `.bak.1` / `.bak.2` / `.bak.3` 备份轮转、坏主档备份读取恢复、旧原型存档迁移、首批内容一致性校验、首批实例与跨块关系校验、地图对象 / 敌人 / 建筑运行时来源复验、配方与区域解锁来源追溯、任务链来源复验、任务目标类型复验、任务目标进度来源与上限复验，以及已完成任务目标满足度校验。后续不要直接跳到复杂云存档或正式多角色 UI，应按以下阶段推进。

#### 阶段 0：第一切片原型

当前阶段保持简单：

- 存档目录使用 Godot `user://saves/slots/slot_01/` 作为当前默认槽位。
- 主存档文件为 `slot_01/slice_01_autosave.json`。
- 覆盖保存前轮转 `slot_01/slice_01_autosave.bak.1.json`、`slot_01/slice_01_autosave.bak.2.json` 和 `slot_01/slice_01_autosave.bak.3.json`。
- 存档内容同时包含 `world` 与 `character`，方便原型闭环和运行时复验。
- 读取时按主存档、`bak.1`、`bak.2`、`bak.3` 依次寻找可用档；全部失败时必须保留当前运行状态，不用坏档覆盖内存中的可玩状态。
- 原型 HUD 已提供 `slot_01`、`slot_02` 和 `slot_03` 三个按钮槽位，显示空槽、可读取、备份可恢复或不可读取状态；`K` / `L` 仍保留为默认 `slot_01` 的调试快捷键。
- 如果默认槽位完全缺失，会尝试读取旧路径 `user://saves/slice_01_autosave.json` 及旧备份，校验成功后写入 `slot_01`，并保留旧文件作为导入来源。

这个阶段只服务第一切片验证，不代表正式存档布局。

#### 阶段 1：首版本地单槽

进入首版前，应把原型单文件升级为“一个世界槽位 + 自动存档 + 备份轮转”：

```text
user://saves/
  worlds/
    world_<stable_id>/
      metadata.json
      autosave.json
      backups/
        autosave.bak.1.json
        autosave.bak.2.json
        autosave.bak.3.json
```

规则：

- `metadata.json` 保存世界显示名、创建时间、更新时间、版本、游玩时长和最近区域等摘要。
- `autosave.json` 是默认进入世界时读取的主文件。
- `backups/` 中保留最近 3 份可读取备份。
- 备份轮转顺序为：写入新主存档前，将 `bak.2` 移为 `bak.3`，`bak.1` 移为 `bak.2`，当前 `autosave.json` 复制为 `bak.1`。
- 如果备份轮转失败，不允许覆盖当前主存档。

#### 阶段 2：首版多槽位

多槽位不应改变“一个世界对应一个存档”的根模型，只是在 `worlds/` 下增加多个世界目录：

```text
user://saves/
  profile/
    local_profile.json
  characters/
    character_<stable_id>.json
  worlds/
    world_<stable_id>/
      metadata.json
      autosave.json
      manual_001.json
      backups/
```

规则：

- 世界进度、基地、区域、任务、建筑、设备和世界库存归属 `worlds/world_<stable_id>/`。
- 角色外观、装备、快捷栏和个人背包先随世界存档保存；等角色档案边界稳定后，再迁出到 `characters/`。
- `local_profile.json` 只保存本地设置、最近进入世界、显示偏好和非关键个人配置。
- 手动存档可以作为同一世界下的额外快照，但不能变成跨世界资源转移入口。

#### 阶段 3：迁移与恢复

正式进入多槽位后，仍应保留从当前 `slot_01` 和旧原型路径导入世界目录的迁移路径：

1. 如果发现 `slice_01_autosave.json`，先按旧格式读取并校验。
2. 读取成功后生成新的 `world_<stable_id>/metadata.json` 和 `autosave.json`。
3. 原旧档移动到 `legacy/` 或保留为只读导入源，不在原路径继续覆盖。
4. 迁移失败时保留旧档，并显示明确失败原因。

坏档恢复优先级：

1. 尝试读取主存档。
2. 主存档缺失或校验失败时，尝试读取 `backups/autosave.bak.1.json`。
3. 继续失败则尝试 `bak.2`、`bak.3`。
4. 所有备份失败时，保留坏档文件，创建安全默认世界，并提示玩家原存档不可读取。

恢复成功后，不应立刻覆盖原坏档；应等玩家再次主动保存或自动保存触发时再写入新的主存档。

### 7.8 联机与云存档边界

首版本地存档不依赖账号、网络或官方服务。

未来接入联机和云存档时，边界应保持：

- 房主开房时，世界存档仍归房主本地或专服。
- 专用服务器使用同一套世界状态结构，但保存位置从 `user://saves/` 迁到服务器数据目录。
- 官方云存档优先同步玩家档案、角色档案和世界备份元数据，不优先实时同步完整世界。
- 云同步必须以本地可读存档为基础，不能让离线单机变成在线依赖。
- 多人世界的冲突合并不在首版处理；Host / 专服是唯一权威写入者。

## 8. 推荐的同步方式

不建议采用帧同步。

推荐方向：

- `服务器权威` 或 `Host 权威`
- `状态同步`
- `客户端插值 / 预测`

原因：

- 你的项目有 ARPG 战斗
- 同时还有建筑、物流、流体、设备运行状态
- 帧同步会让调试和扩展难度显著增加
- 状态同步更适合这类混合型项目

## 9. 为什么不建议 P2P 作为主方案

虽然 P2P 听起来更轻，但不适合作为你的核心方向。

原因：

- 自动化基地状态复杂
- 战斗判定需要统一
- 掉线重连麻烦
- 不同步和作弊问题更明显
- 后续迁移到专用服务器时路径不顺

因此更稳的方向是：

- 单机：本地权威
- 开房：Host 权威
- 专服：服务器权威

逻辑模型保持统一，只是权威位置不同。

## 10. 现在就必须遵守的架构原则

这部分最重要。

即使现在完全不做联机，代码结构上也必须提前遵守这些原则。

### 10.1 游戏逻辑与表现层分离

不要把以下内容直接写死在节点表现逻辑里：

- 战斗结算
- 背包变化
- 资源扣除
- 建造结果
- 设备运行状态
- 掉落生成
- 任务推进

正确做法是：

- 表现层负责显示
- 系统层负责规则
- 数据层负责状态

### 10.2 所有核心对象都必须有稳定唯一 ID

至少包括：

- 玩家
- 怪物
- NPC
- 建筑
- 设备
- 掉落物
- 子弹 / 技能实例
- 资源点
- 储罐
- 管道节点

没有稳定 ID，后期同步和存档都会很难做。

### 10.3 所有关键交互尽量命令化

推荐把这些行为设计成命令：

- 移动
- 攻击
- 释放技能
- 使用道具
- 建造
- 拆除
- 启动设备
- 停止设备
- 向容器放入资源
- 从容器取出资源
- 接受任务
- 提交任务

这样以后无论是单机、本地主机还是服务器，都能复用同一套交互模型。

### 10.4 数据必须可序列化

以下状态必须从一开始就按可保存、可加载、可同步的方式设计：

- 玩家状态
- 背包
- 装备
- 技能冷却
- 任务状态
- 建筑状态
- 设备状态
- 资源库存
- 污染状态
- 区域状态
- 世界事件状态

### 10.5 单机也按“本地模拟服务器”心智设计

即便首版只有单机，也建议在概念上拆成：

- 输入
- 命令处理
- 状态更新
- 表现刷新
- 存档写入

这样以后接入联机时，不需要重新推翻核心循环。

## 11. 2~5 人协作与 2~10 人扩展边界

既然当前首版设计基准是 2~5 人，很多设计就应该按这个范围考虑。

2~10 人只作为长期专服或高性能房间的扩展目标，不作为首版体验基准。

### 11.1 玩法上适合的内容

- 小队远征
- 分工建设
- 资源运输协作
- 一人负责基地，一人负责外勤
- 多人协作处理污染和事故
- 多人打 Boss
- 不同角色天赋形成轻量分工

### 11.2 不适合提前做大的内容

- 大规模职业平衡
- 团本式 10 人以上机制设计
- 超复杂仇恨和团队技能体系
- 大型组织管理玩法

### 11.3 推荐的角色协作方向

可以考虑后期自然形成这些倾向，但不要首版做复杂职业锁死：

- 输出倾向
- 工程倾向
- 生存 / 支援倾向
- 采集 / 外勤倾向

这些倾向可以来自角色初始天赋、装备模块、技能选择和基地生产，而不是由固定职业完全决定。

## 12. 世界进入与权限模型建议

这部分后面很容易出问题，建议现在就先定原则。

### 12.1 世界进入

允许的方式：

- 离线进入自己的世界
- 好友直接加入房主世界
- 通过邀请码 / 房间码加入
- 后续可通过服务器列表加入专服

### 12.2 权限控制

进入别人世界时，最好预留权限等级：

- 访客：只能探索和战斗
- 协作者：可采集、可建造有限内容
- 管理者：可改基地、可拆除、可调整配置
- 房主 / 服主：完全控制

这样后期多人共建时不容易失控。

## 13. 云存档与官方服务应该怎么做

官方云存档和官方服务是中后期能力，不应前置为首版条件。

推荐顺序：

1. 本地存档
2. 房间联机
3. 专服持久化
4. 账号系统
5. 云存档同步
6. 官方服务器目录 / 邀请体系

### 13.1 官方云存档建议保存什么

- 玩家档案
- 角色档案
- 成就
- 设置
- 可选的世界备份元数据

### 13.2 不建议过早云化的内容

- 全部世界实时同步
- 全服统一市场
- 全局共享基地资源

## 14. 推荐的长期架构演进路线

### 阶段一：纯单机

- 本地世界存档
- 本地角色档案
- 核心玩法完整成立

### 阶段二：本地主机开房

- 2~5 人加入
- 房主权威
- 基础同步
- 玩家可以选择角色加入房主世界
- 房主可以限制外部角色、背包和装备携带

### 阶段三：专用服务器

- 长期世界运行
- 玩家随时加入
- 服务器保存世界状态
- 在性能和同步稳定后评估 2~10 人上限

### 阶段四：官方账号与云存档

- 登录
- 角色云档
- 服务器列表
- 好友邀请

## 15. 现在开发时必须避免的坑

- 不要把所有状态塞在 Godot 场景节点临时变量里
- 不要让 UI 直接修改游戏核心状态
- 不要把单机逻辑写成完全不可序列化的对象图
- 不要让建筑和设备没有稳定 ID
- 不要先做依赖网络的核心玩法
- 不要在首版为了远期 MMO 引入过重系统

## 16. 当前阶段的执行建议

现在虽然还不做联机，但建议马上落实这些内容：

- 定义世界存档结构
- 定义玩家档案结构
- 定义角色档案结构
- 定义对象 ID 规则
- 定义关键命令模型
- 定义本地存档格式
- 在系统设计时强制区分“规则层”和“表现层”

## 17. 最终结论

本项目的联机方向应正式确定为：

**单机离线可玩 + 本地开房协作 + 可迁移到专用服务器 + 后续可接官方账号和云存档的 Minecraft 式多形态联机模型。**

协作人数方向为：

**首版按 2~5 人协作设计，长期预留 2~10 人专服或高性能房间扩展；更远期的大型多人在线可以作为方向保留，但不进入首版和首个联机版本的实现范围。**

角色和存档方向为：

**一个玩家可以拥有多个角色，不同角色可以有不同初始能力和天赋；不同世界存档可以选择不同角色进入，但世界进度、基地资源、关键装备和主线推进应默认归属世界存档，并由房主或服务器规则控制跨世界携带。**

而为了未来不重构，现在开发时最重要的不是立刻做联机，而是：

**从第一天开始按“同一套世界规则可运行在单机、Host 房间和专用服务器”去设计状态结构、交互模型和代码边界。**
