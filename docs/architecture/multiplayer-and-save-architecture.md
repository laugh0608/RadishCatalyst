# RadishCatalyst / 异星催化 联机模式与存档架构前置设计

## 1. 文档目的

这份文档用于在项目正式开始开发前，提前锁定联机模式、存档归属、同步方式和架构预留原则。

项目正式名称：

- 英文名：`RadishCatalyst`
- 中文名：`异星催化`

目标不是现在立刻实现完整联机，而是：

- 提前决定联机的最终方向
- 避免前期按纯单机写死，后期大改甚至重构
- 为客户端、数据结构、系统边界和后端接入方式设定约束

这份文档应被视为后续功能设计和代码结构设计的前置原则文档。

## 2. 最终联机方向

本项目不走 MMO 大世界路线。

本项目的联机目标正式定义为：

**类似《我的世界》的多形态联机模型：支持离线单机、本地存档、玩家开房协作、专用服务器长期存档，以及后续可选的官方账号与官方云存档。**

当前建议的协作人数范围：

- `2 ~ 5 人`

这个方向的核心价值是：

- 单机先成立
- 联机后扩展体验
- 不把项目一开始拖入 MMO 级复杂度
- 同时保留长期商业化和官方服务扩展空间

## 3. 联机形态分层

项目应按四层联机形态设计。

### 3.1 第一层：离线单机

特点：

- 本地运行
- 本地存档
- 不依赖网络
- 所有核心玩法完整可玩

要求：

- 没有网络时游戏也必须成立
- 所有核心玩法不能依赖在线服务
- 单机模式是整个项目的最小闭环基础

### 3.2 第二层：玩家开房 / 局域网 / 好友联机

特点：

- 一名玩家作为 Host
- 其他玩家加入房间
- 适合 2~5 人协作
- 以房主存档作为世界状态来源

适用玩法：

- 协作探索
- 协作战斗
- 小队远征
- 基地共同建设
- Boss 战

### 3.3 第三层：专用服务器

特点：

- 玩家自建或租用服务器
- 服务器长期保存世界状态
- 多人可以在固定世界中持续协作

适用目标：

- 固定合作档
- 稳定队伍长期推进
- 避免房主不上线时无法进入存档

### 3.4 第四层：官方服务

特点：

- 官方账号体系
- 官方云存档
- 官方服务器列表
- 好友与邀请系统
- 官方服务器或官方中继服务

说明：

- 这是中后期扩展目标
- 不是首版开发前提
- 不能成为前期核心玩法成立的依赖条件

## 4. 为什么不走 MMO 大世界

不建议把这个项目定义成 MMORPG，原因如下：

- 自动化基地状态同步本身就很复杂
- ARPG 战斗已经有较高实时同步要求
- 大世界多人在线会放大网络、存档、权限、反作弊和运维压力
- 个人开发者很难在首版同时做好基地模拟、实时战斗和 MMO 级在线系统

因此本项目的联机目标更适合定义为：

**房间制 / 专服制的协作联机 ARPG，而不是 MMO。**

## 5. 推荐的联机核心玩法边界

联机应该优先增强这些体验：

- 组队外出探索
- 协作清理高危区域
- 一起打精英怪和 Boss
- 分工采集与运回资源
- 在同一基地内分工建设和维护
- 协作完成高阶工业链和区域推进

联机不应该优先做这些内容：

- 大规模 PvP
- 开放式自由交易经济
- 大型公会战
- 全服共享世界资源刷新
- 长期数百人同屏公共地图

## 6. 推荐的世界与房间模型

最稳妥的模型是：

- 一个世界对应一个存档
- 一个存档可以离线单机游玩
- 该存档也可以被房主作为联机房间打开
- 后续同一个存档也可以迁移到专用服务器

这意味着世界的本质应该是：

- 同一份规则
- 同一套状态结构
- 不同运行方式

也就是：

- 单机运行时：本地模拟 + 本地存档
- Host 联机时：本地主机模拟 + 远程玩家加入
- 专服联机时：服务器模拟 + 客户端连接

这是整个项目未来避免重构的关键。

## 7. 存档归属模型

建议明确区分两类存档：

### 7.1 世界存档

用于保存：

- 地图进度
- 建筑布局
- 设备状态
- 管网状态
- 储罐内容
- 电网状态
- 区域解锁
- 主线推进
- 世界任务状态
- 基地库存

它代表的是“这个世界本身”。

### 7.2 玩家档案

用于保存：

- 玩家角色数据
- 等级
- 技能
- 装备
- 快捷栏
- 个人背包
- 外观
- 好友与账号信息

它代表的是“这个玩家自己”。

### 7.3 单机模式下的关系

单机时，世界存档和玩家档案可以一起保存在本地。

### 7.4 开房模式下的关系

开房时：

- 世界存档归房主
- 其他玩家带入自己的角色档案，或使用房间规则指定的临时角色

建议优先考虑两种模式中的一种：

1. 角色绑定世界  
进入某个世界时，角色成长跟随该世界推进

2. 角色独立于世界  
角色可以进入不同世界，但要处理平衡和资源携带问题

从开发复杂度看，首版建议优先：

**角色与世界弱绑定，但跨世界资源和装备携带要严格限制。**

## 8. 推荐的同步方式

不建议采用帧同步。

推荐方向：

- `服务器权威` 或 `Host 权威`
- `状态同步`
- `客户端插值 / 预测`

原因：

- 你的项目有 ARPG 战斗
- 同时还有建筑、物流、流体、设备运行状态
- 帧同步会让调试和扩展难度显著增加
- 状态同步更适合这类混合型项目

## 9. 为什么不建议 P2P 作为主方案

虽然 P2P 听起来更轻，但不适合作为你的核心方向。

原因：

- 自动化基地状态复杂
- 战斗判定需要统一
- 掉线重连麻烦
- 不同步和作弊问题更明显
- 后续迁移到专用服务器时路径不顺

因此更稳的方向是：

- 单机：本地权威
- 开房：Host 权威
- 专服：服务器权威

逻辑模型保持统一，只是权威位置不同。

## 10. 现在就必须遵守的架构原则

这部分最重要。

即使现在完全不做联机，代码结构上也必须提前遵守这些原则。

### 10.1 游戏逻辑与表现层分离

不要把以下内容直接写死在节点表现逻辑里：

- 战斗结算
- 背包变化
- 资源扣除
- 建造结果
- 设备运行状态
- 掉落生成
- 任务推进

正确做法是：

- 表现层负责显示
- 系统层负责规则
- 数据层负责状态

### 10.2 所有核心对象都必须有稳定唯一 ID

至少包括：

- 玩家
- 怪物
- NPC
- 建筑
- 设备
- 掉落物
- 子弹 / 技能实例
- 资源点
- 储罐
- 管道节点

没有稳定 ID，后期同步和存档都会很难做。

### 10.3 所有关键交互尽量命令化

推荐把这些行为设计成命令：

- 移动
- 攻击
- 释放技能
- 使用道具
- 建造
- 拆除
- 启动设备
- 停止设备
- 向容器放入资源
- 从容器取出资源
- 接受任务
- 提交任务

这样以后无论是单机、本地主机还是服务器，都能复用同一套交互模型。

### 10.4 数据必须可序列化

以下状态必须从一开始就按可保存、可加载、可同步的方式设计：

- 玩家状态
- 背包
- 装备
- 技能冷却
- 任务状态
- 建筑状态
- 设备状态
- 资源库存
- 污染状态
- 区域状态
- 世界事件状态

### 10.5 单机也按“本地模拟服务器”心智设计

即便首版只有单机，也建议在概念上拆成：

- 输入
- 命令处理
- 状态更新
- 表现刷新
- 存档写入

这样以后接入联机时，不需要重新推翻核心循环。

## 11. 2~5 人协作的设计边界

既然当前暂定人数是 2~5 人，很多设计就应该按这个范围考虑。

### 11.1 玩法上适合的内容

- 小队远征
- 分工建设
- 资源运输协作
- 一人负责基地，一人负责外勤
- 多人协作处理污染和事故
- 多人打 Boss

### 11.2 不适合提前做大的内容

- 大规模职业平衡
- 团本式 10 人以上机制设计
- 超复杂仇恨和团队技能体系
- 大型组织管理玩法

### 11.3 推荐的角色协作方向

可以考虑后期自然形成这些倾向，但不要首版做复杂职业锁死：

- 输出倾向
- 工程倾向
- 生存 / 支援倾向
- 采集 / 外勤倾向

## 12. 世界进入与权限模型建议

这部分后面很容易出问题，建议现在就先定原则。

### 12.1 世界进入

允许的方式：

- 离线进入自己的世界
- 好友直接加入房主世界
- 通过邀请码 / 房间码加入
- 后续可通过服务器列表加入专服

### 12.2 权限控制

进入别人世界时，最好预留权限等级：

- 访客：只能探索和战斗
- 协作者：可采集、可建造有限内容
- 管理者：可改基地、可拆除、可调整配置
- 房主 / 服主：完全控制

这样后期多人共建时不容易失控。

## 13. 云存档与官方服务应该怎么做

官方云存档和官方服务是中后期能力，不应前置为首版条件。

推荐顺序：

1. 本地存档
2. 房间联机
3. 专服持久化
4. 账号系统
5. 云存档同步
6. 官方服务器目录 / 邀请体系

### 13.1 官方云存档建议保存什么

- 玩家档案
- 角色成长
- 成就
- 设置
- 可选的世界备份元数据

### 13.2 不建议过早云化的内容

- 全部世界实时同步
- 全服统一市场
- 全局共享基地资源

## 14. 推荐的长期架构演进路线

### 阶段一：纯单机

- 本地世界存档
- 本地角色档案
- 核心玩法完整成立

### 阶段二：本地主机开房

- 2~5 人加入
- 房主权威
- 基础同步

### 阶段三：专用服务器

- 长期世界运行
- 玩家随时加入
- 服务器保存世界状态

### 阶段四：官方账号与云存档

- 登录
- 角色云档
- 服务器列表
- 好友邀请

## 15. 现在开发时必须避免的坑

- 不要把所有状态塞在 Godot 场景节点临时变量里
- 不要让 UI 直接修改游戏核心状态
- 不要把单机逻辑写成完全不可序列化的对象图
- 不要让建筑和设备没有稳定 ID
- 不要先做依赖网络的核心玩法
- 不要先为了“未来 MMO”引入过重系统

## 16. 当前阶段的执行建议

现在虽然还不做联机，但建议马上落实这些内容：

- 定义世界存档结构
- 定义玩家档案结构
- 定义对象 ID 规则
- 定义关键命令模型
- 定义本地存档格式
- 在系统设计时强制区分“规则层”和“表现层”

## 17. 最终结论

本项目的联机方向应正式确定为：

**单机离线可玩 + 本地开房协作 + 可迁移到专用服务器 + 后续可接官方账号和云存档的 Minecraft 式多形态联机模型。**

而为了未来不重构，现在开发时最重要的不是立刻做联机，而是：

**从第一天开始按“同一套世界规则可运行在单机、Host 房间和专用服务器”去设计状态结构、交互模型和代码边界。**
