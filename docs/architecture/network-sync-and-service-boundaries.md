## 8. 推荐的同步方式

不建议采用帧同步。

推荐方向：

- `服务器权威` 或 `Host 权威`
- `状态同步`
- `客户端插值 / 预测`

原因：

- 你的项目有 ARPG 战斗
- 同时还有建筑、物流、流体、设备运行状态
- 帧同步会让调试和扩展难度显著增加
- 状态同步更适合这类混合型项目

## 9. 为什么不建议 P2P 作为主方案

虽然 P2P 听起来更轻，但不适合作为你的核心方向。

原因：

- 自动化基地状态复杂
- 战斗判定需要统一
- 掉线重连麻烦
- 不同步和作弊问题更明显
- 后续迁移到专用服务器时路径不顺

因此更稳的方向是：

- 单机：本地权威
- 开房：Host 权威
- 专服：服务器权威

逻辑模型保持统一，只是权威位置不同。

## 10. 现在就必须遵守的架构原则

这部分最重要。

即使现在完全不做联机，代码结构上也必须提前遵守这些原则。

### 10.1 游戏逻辑与表现层分离

不要把以下内容直接写死在节点表现逻辑里：

- 战斗结算
- 背包变化
- 资源扣除
- 建造结果
- 设备运行状态
- 掉落生成
- 任务推进

正确做法是：

- 表现层负责显示
- 系统层负责规则
- 数据层负责状态

### 10.2 所有核心对象都必须有稳定唯一 ID

至少包括：

- 玩家
- 怪物
- NPC
- 建筑
- 设备
- 掉落物
- 子弹 / 技能实例
- 资源点
- 储罐
- 管道节点

没有稳定 ID，后期同步和存档都会很难做。

### 10.3 所有关键交互尽量命令化

推荐把这些行为设计成命令：

- 移动
- 攻击
- 释放技能
- 使用道具
- 建造
- 拆除
- 启动设备
- 停止设备
- 向容器放入资源
- 从容器取出资源
- 接受任务
- 提交任务

这样以后无论是单机、本地主机还是服务器，都能复用同一套交互模型。

### 10.4 数据必须可序列化

以下状态必须从一开始就按可保存、可加载、可同步的方式设计：

- 玩家状态
- 背包
- 装备
- 技能冷却
- 任务状态
- 建筑状态
- 设备状态
- 资源库存
- 污染状态
- 区域状态
- 世界事件状态

### 10.5 单机也按“本地模拟服务器”心智设计

即便首版只有单机，也建议在概念上拆成：

- 输入
- 命令处理
- 状态更新
- 表现刷新
- 存档写入

这样以后接入联机时，不需要重新推翻核心循环。

## 11. 2~5 人协作与 2~10 人扩展边界

既然当前首版设计基准是 2~5 人，很多设计就应该按这个范围考虑。

2~10 人只作为长期专服或高性能房间的扩展目标，不作为首版体验基准。

### 11.1 玩法上适合的内容

- 小队远征
- 分工建设
- 资源运输协作
- 一人负责基地，一人负责外勤
- 多人协作处理污染和事故
- 多人打 Boss
- 不同角色天赋形成轻量分工

### 11.2 不适合提前做大的内容

- 大规模职业平衡
- 团本式 10 人以上机制设计
- 超复杂仇恨和团队技能体系
- 大型组织管理玩法

### 11.3 推荐的角色协作方向

可以考虑后期自然形成这些倾向，但不要首版做复杂职业锁死：

- 输出倾向
- 工程倾向
- 生存 / 支援倾向
- 采集 / 外勤倾向

这些倾向可以来自角色初始天赋、装备模块、技能选择和基地生产，而不是由固定职业完全决定。

## 12. 世界进入与权限模型建议

这部分后面很容易出问题，建议现在就先定原则。

### 12.1 世界进入

允许的方式：

- 离线进入自己的世界
- 好友直接加入房主世界
- 通过邀请码 / 房间码加入
- 后续可通过服务器列表加入专服

### 12.2 权限控制

进入别人世界时，最好预留权限等级：

- 访客：只能探索和战斗
- 协作者：可采集、可建造有限内容
- 管理者：可改基地、可拆除、可调整配置
- 房主 / 服主：完全控制

这样后期多人共建时不容易失控。

## 13. 云存档与官方服务应该怎么做

官方云存档和官方服务是中后期能力，不应前置为首版条件。

推荐顺序：

1. 本地存档
2. 房间联机
3. 专服持久化
4. 账号系统
5. 云存档同步
6. 官方服务器目录 / 邀请体系

### 13.1 官方云存档建议保存什么

- 玩家档案
- 角色档案
- 成就
- 设置
- 可选的世界备份元数据

### 13.2 不建议过早云化的内容

- 全部世界实时同步
- 全服统一市场
- 全局共享基地资源
