# Save Data Model - 世界存档结构

返回：[Save Data Model](save-data-model.md)

## 世界存档结构建议

首版世界存档可以按逻辑块组织：

```text
world
metadata
regions
map_cells
map_objects
base
inventories
pollution
time_weather
quests
events
```

### metadata

保存：

- 存档版本。
- 游戏版本。
- 世界 ID。
- 世界显示名。
- 创建时间。
- 最近保存时间。
- 最近使用角色 ID。

### regions

保存区域级状态：

- 区域 ID。
- 是否已发现。
- 是否已解锁。
- 探索比例或关键节点进度。
- 危险等级修正。
- 敌对生态活跃度。
- 区域目标状态。

### map_cells

保存地块级状态。

首版建议使用离散地块或网格心智，即使实际 Godot 表现不是严格网格，也应有可序列化的地块数据。

保存：

- 坐标。
- 高度等级。
- 地表类型。
- 是否已清理。
- 是否已平整。
- 是否有地板或地基。
- 地基等级。
- 可建造性覆盖。
- 污染值或污染类型引用。
- 是否已探索。

不建议保存：

- 每帧视觉效果。
- 可由地块状态推导出的装饰细节。
- 临时粒子、脚印或短期战斗痕迹。

### map_objects

保存地图对象实例。

对象包括：

- 资源节点。
- 样本节点。
- 可破坏障碍。
- 不可破坏但需要记录发现状态的遗迹节点。
- 敌人巢穴。
- 掉落物。
- 任务交互点。

保存：

- 实例 ID。
- 定义 ID。
- 位置。
- 朝向或形态。
- 当前状态，例如完整、受损、已破坏、已采集、已采样、已净化。
- 当前耐久或剩余资源量。
- 是否已被玩家发现。
- 关联任务或区域事件。

原则：

- 不可破坏对象通常不需要保存“耐久”，但可能需要保存是否发现、是否激活、是否解锁。
- 可破坏对象必须保存当前状态，避免加载后复原。
- 可采集对象应保存剩余量或已采集状态。
- 锚场回稳这类固定交互点可保存部署、压力和稳定完成标记，用于读档后恢复已部署校锚桩、守脉体压力和局部稳定窗口完成态。

### base

保存基地和工业系统状态。

包括：

- 基地核心。
- 建筑和设备实例。
- 管线、泵、储罐和连接关系。
- 电力或能源网络。
- 设备运行状态。
- 设备配方选择。
- 设备输入、输出和内部缓冲。
- 设备故障、堵塞、过热或污染超标状态。

设备实例保存：

- 实例 ID。
- 定义 ID。
- 位置和占地。
- 朝向。
- 所在地基或地块引用。
- 当前配方 ID。
- 运行状态。
- 处理进度。
- 内部库存。
- 能源需求和当前供给状态。
- 异常状态。

### inventories

保存库存状态。

库存类型：

- 基地总库存。
- 设备内部缓存。
- 箱体。
- 储罐。
- 玩家背包。
- 地面掉落。

物品库存保存：

- 物品 ID。
- 数量。
- 可选品质或耐久。

流体库存保存：

- 流体 ID。
- 数量。
- 可选纯度、温度、污染度或稳定性。

首版可以先不做复杂品质、温度和压力，但字段位置应预留。

### pollution

保存污染状态。

建议分层：

- 区域污染总览。
- 地块污染值。
- 污染源点状态。
- 污染处理设施状态。
- 污染事件或生态反应计数。

污染字段：

- 污染类型。
- 当前数值。
- 趋势或最近变化。
- 来源引用。
- 是否已治理。
- 对敌人、建造或防护消耗的影响标记。

### time_weather

保存时间与天气状态。

首版至少预留：

- 世界时间。
- 日期或周期计数。
- 当前时间段，例如白天、夜晚。
- 当前天气 ID。
- 天气剩余时间。
- 下一个天气候选或随机种子。
- 区域天气覆盖。

原则：

- 天气必须可保存，避免加载后随机跳变。
- 天气影响应可由天气 ID 和区域状态推导。
- 不做复杂气候模拟时，也应保留扩展字段。

### quests

保存任务和解锁。

包括：

- 主线阶段。
- 区域目标。
- 已完成教程提示。
- 已解锁配方。
- 已解锁设备。
- 已解锁区域入口。
- 已分析样本。

### events

保存世界事件。

包括：

- 已触发但未完成的事件。
- 已完成事件。
- 敌袭或污染反应计时。
- 秘境入口状态。
- 事故记录或当前事故。

首版可以简单，但不应把事件状态散落在场景节点里。
