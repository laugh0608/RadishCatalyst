# RadishCatalyst / 异星催化 联机模式与存档架构前置设计 - 联机模型

返回：[RadishCatalyst / 异星催化 联机模式与存档架构前置设计](multiplayer-and-save-architecture.md)

## 1. 文档目的

这份文档用于在项目正式开始开发前，提前锁定联机模式、存档归属、同步方式和架构预留原则。

项目正式名称：

- 英文名：`RadishCatalyst`
- 中文名：`异星催化`

目标不是现在立刻实现完整联机，而是：

- 提前决定联机的最终方向
- 避免前期按纯单机写死，后期大改甚至重构
- 为客户端、数据结构、系统边界和后端接入方式设定约束

这份文档应被视为后续功能设计和代码结构设计的前置原则文档。

## 2. 最终联机方向

本项目在首版和首个存档 + 联机版本中不按 MMO 大世界路线设计。

远期可以保留大型多人在线的演进可能，但它不是当前阶段、首版或首个联机版本的设计前提。

本项目的联机目标正式定义为：

**类似《我的世界》的多形态联机模型：支持离线单机、本地存档、玩家开房协作、专用服务器长期存档，以及后续可选的官方账号与官方云存档。**

当前建议的协作人数范围：

- 首版设计基准：`2 ~ 5 人`
- 长期可扩展上限：`2 ~ 10 人`

这个方向的核心价值是：

- 单机先成立
- 联机后扩展体验
- 不把项目一开始拖入 MMO 级复杂度
- 同时保留长期商业化和官方服务扩展空间

人数口径的含义：

- `2 ~ 5 人` 是玩法、UI、平衡和同步复杂度的默认设计目标。
- `2 ~ 10 人` 是专用服务器或高性能主机房间的远期扩展上限，不作为首版承诺。
- 不应为了支持 10 人而提前设计团本、职业铁三角或复杂团队调度。

## 3. 联机形态分层

项目应按四层联机形态设计。

### 3.1 第一层：离线单机

特点：

- 本地运行
- 本地存档
- 不依赖网络
- 所有核心玩法完整可玩

要求：

- 没有网络时游戏也必须成立
- 所有核心玩法不能依赖在线服务
- 单机模式是整个项目的最小闭环基础

### 3.2 第二层：玩家开房 / 局域网 / 好友联机

特点：

- 一名玩家作为 Host
- 其他玩家加入房间
- 适合 2~5 人协作
- 以房主存档作为世界状态来源
- 远期可以在性能和同步验证后扩展到 2~10 人

适用玩法：

- 协作探索
- 协作战斗
- 小队远征
- 基地共同建设
- Boss 战

### 3.3 第三层：专用服务器

特点：

- 玩家自建或租用服务器
- 服务器长期保存世界状态
- 多人可以在固定世界中持续协作

适用目标：

- 固定合作档
- 稳定队伍长期推进
- 避免房主不上线时无法进入存档

### 3.4 第四层：官方服务

特点：

- 官方账号体系
- 官方云存档
- 官方服务器列表
- 好友与邀请系统
- 官方服务器或官方中继服务

说明：

- 这是中后期扩展目标
- 不是首版开发前提
- 不能成为前期核心玩法成立的依赖条件

## 4. 为什么当前不按 MMO 大世界设计

不建议在首版或首个存档 + 联机版本中把这个项目定义成 MMORPG，原因如下：

- 自动化基地状态同步本身就很复杂
- ARPG 战斗已经有较高实时同步要求
- 大世界多人在线会放大网络、存档、权限、反作弊和运维压力
- 个人开发者很难在首版同时做好基地模拟、实时战斗和 MMO 级在线系统

因此本项目的联机目标更适合定义为：

**当前阶段先做房间制 / 专服制的协作联机 ARPG，并为更远期的大型多人在线保留架构口子。**

## 5. 推荐的联机核心玩法边界

联机应该优先增强这些体验：

- 组队外出探索
- 协作清理高危区域
- 一起打精英怪和 Boss
- 分工采集与运回资源
- 在同一基地内分工建设和维护
- 协作完成高阶工业链和区域推进

联机不应该优先做这些内容：

- 大规模 PvP
- 开放式自由交易经济
- 大型公会战
- 全服共享世界资源刷新
- 长期数百人同屏公共地图

## 6. 推荐的世界与房间模型

最稳妥的模型是：

- 一个世界对应一个存档
- 一个存档可以离线单机游玩
- 该存档也可以被房主作为联机房间打开
- 后续同一个存档也可以迁移到专用服务器

这意味着世界的本质应该是：

- 同一份规则
- 同一套状态结构
- 不同运行方式

也就是：

- 单机运行时：本地模拟 + 本地存档
- Host 联机时：本地主机模拟 + 远程玩家加入
- 专服联机时：服务器模拟 + 客户端连接

这是整个项目未来避免重构的关键。
