# Platform And Compatibility

更新时间：2026-04-26

## 文档目的

这份文档用于在正式初始化 Godot 工程和拆 MVP 功能之前，先明确 RadishCatalyst 的首发平台、发布顺序、Web 定位、离线单机要求、存档原则、脚本语言边界和长期兼容边界。

它不是构建脚本说明，也不是发行计划。它只固定上位技术方向，避免前期做出会阻碍未来 Web、Android、联机、Wiki 和官方工具的结构选择。

## 平台优先级结论

当前平台口径为：

1. Windows 桌面版：核心游戏主平台和首版主目标。
2. Web 试玩 Demo / 技术验证：可选，不作为 MVP 必须交付。
3. Android：远期评估，不作为首版约束。
4. WebApp：值得规划，但主要指 Wiki、配方查询器、工艺链规划器、地图 / 资源工具，不是完整游戏客户端。
5. 其他平台暂不纳入当前决策。

首发产品形态优先考虑 Windows。本项目的主要操作、信息密度、基地规划和 ARPG 战斗都更适合鼠标键盘与桌面屏幕。

Web 游戏客户端只适合传播、试玩和小范围验证，不应在首版阶段反向限制核心体验，也不应成为技术选型的主约束。

Android 只做方向预留，不作为当前首版目标。

## Windows 桌面版

Windows 是首发客户端优先级最高的平台。

原因：

- Godot 4.x 对 Windows 桌面导出成熟。
- 鼠标键盘适合基地建设、设备面板、地图反查和战斗操作。
- 本地文件存档、日志和调试更直接。
- 长期更接近 Steam 等桌面发行路径。

Windows 版本必须支持：

- 离线单机。
- 本地存档。
- 本地设置。
- 可导入 / 导出或备份存档的结构预留。
- 后续接入房间联机、专用服务器或官方服务时的兼容路径。

## Web 试玩 Demo / 技术验证

Web 游戏客户端的定位是试玩和传播，不是首版完整体验，也不是 MVP 必须交付项。

适合承载：

- 5 到 10 分钟轻量 Demo。
- 核心操作验证。
- 小规模战斗或基地片段展示。
- 官网、活动页或社区传播。
- 技术展示和玩家反馈收集。

不适合承诺：

- 完整存档体验。
- 大地图长期游玩。
- 高复杂基地。
- 与 Windows 完全一致的功能。
- 作为首版开发门槛。

Web 版本需要注意：

- 文件系统能力受限。
- 性能、内存和加载包大小更敏感。
- 多线程、网络、文件持久化和浏览器兼容性可能限制完整玩法。
- 复杂本地存档需要封装，不应直接依赖桌面路径。

因此从数据和存档设计上要预留抽象层，但不要因为 Web 限制而牺牲 Windows 主体验。

如果 Web 试玩与 Windows 主体验发生冲突，优先保证 Windows 主体验。

## WebApp 定位

WebApp 是重要方向，但它和完整游戏 Web 客户端不是一回事。

WebApp 优先指：

- 玩家 Wiki。
- 在线合成台。
- 配方查询器。
- 工艺链规划器。
- 地图 / 资源标点工具。
- 怪物掉落和装备 / 药剂查询工具。

这些工具应优先复用结构化游戏数据，服务玩家查询、规划和社区传播。

WebApp 可以比完整游戏客户端更早进入规划，但不应被误解为“必须把完整游戏做成 Web 端”。

## Android 远期评估

Android 暂不作为首发目标。

保留它的原因：

- 后续可能适合轻量试玩、辅助工具或简化客户端。
- 官方工具、Wiki 和合成台天然适合移动端访问。
- 如果核心游戏后续支持触控，Android 可成为扩展平台。

当前不承诺：

- 首版 Android 客户端。
- 与桌面完全一致的操作体验。
- 在小屏幕上承载完整基地规划和高密度设备面板。

当前需要预留：

- UI 不要完全依赖极小点击目标。
- 核心数据结构和工具页面适合响应式展示。
- 输入层不要把鼠标键盘逻辑写死进核心规则。

Android 不应反向约束首版 Windows 操作密度、基地面板复杂度和本地存档设计。

## GDScript 与 .NET / C# 边界

当前 Godot 客户端首版默认使用 GDScript。

原因：

- GDScript 更贴合 Godot 场景、节点、信号和 UI 工作流。
- 首版重点是验证可玩闭环，不是先建立重型工程栈。
- Windows、Web 试玩和 Android 远期评估的共同基线更适合先用 GDScript。
- Godot 4 的 C# / .NET 路线会引入额外工具链和平台兼容风险，尤其不适合作为 Web 游戏客户端核心依赖。

.NET / C# 仍然是重要能力，但边界应放在：

- 外部数据校验工具。
- 配方和静态数据编辑 / 检查工具。
- 存档迁移工具。
- 官方 WebApp 后端或构建工具。
- 后续服务端、房间服务或专用服务器。
- 未来非 Web 客户端中的局部高复杂规则模块，前提是先重新评估平台影响。

当前不做：

- 不把 Godot 客户端核心逻辑押到 C#。
- 不因为团队熟悉 .NET 而牺牲 Web 试玩可选性和 Godot 原型速度。
- 不在没有明确收益前引入混合脚本复杂度。

如果未来决定引入 C#，必须先完成一次平台兼容复核，并同步更新：

- `platform-and-compatibility.md`
- `godot-project-structure.md`
- `save-data-model.md`
- 当前阶段计划和周志。

## 离线单机原则

首版必须离线单机可玩。

含义：

- 没有网络也能进入游戏。
- 核心循环不依赖账号、服务器、云存档或在线验证。
- 本地世界存档和玩家档案足以支撑完整进度。
- 官方服务失败时，不影响单机游玩。

这条原则优先级高于联机扩展。

## 本地存档原则

本地存档必须从一开始按长期兼容设计。

建议至少区分：

- 世界存档：地图、基地、设备、区域、任务、污染和世界事件。
- 玩家档案：角色等级、装备、技能、背包、快捷栏、设置和个人进度。
- 全局设置：图像、音频、输入、语言、辅助功能等。

存档设计原则：

- 有明确版本号。
- 可迁移。
- 可校验。
- 可备份。
- 尽量避免直接序列化不可控对象图。
- 核心状态以结构化数据表达，而不是散落在场景节点临时变量中。

存档路径和具体格式可在工程初始化后再定，但当前文档层面必须先确定：本地存档是首版核心能力，不是后续补丁。

## 未来联机兼容

本项目长期联机方向以 `multiplayer-and-save-architecture.md` 为准：

- 离线单机。
- 玩家开房 / 好友联机。
- 专用服务器。
- 后续可选官方账号与云存档。

为了兼容这些形态，当前单机设计也应保持以下原则：

- 单机世界仍按“本地权威”心智运行。
- 核心交互尽量命令化。
- 游戏规则、状态和表现层分离。
- 世界状态和玩家状态边界清楚。
- 不把服务器、账号或云服务写成核心玩法前提。

## 数据 ID 原则

所有可被存档、同步、反查或公开展示的对象，都应有稳定 ID。

至少包括：

- 物品。
- 材料。
- 配方。
- 设备。
- 建筑。
- 区域。
- 资源点。
- 怪物。
- 掉落表。
- 技能。
- 装备。
- 任务。
- Wiki 条目。
- 官方工具查询对象。

稳定 ID 原则：

- 不使用显示名作为唯一标识。
- ID 使用英文、稳定、可读的命名。
- 显示名、描述和本地化文本与 ID 分离。
- ID 一旦进入存档或公开数据，应避免随意改名。
- 必须改名时，需要迁移规则或别名表。

## 结构化数据原则

RadishCatalyst 天然包含大量可查询关系，因此数据不能只写在散文档里。

未来应尽量结构化的内容：

- 物品与分类。
- 配方输入、输出和副产物。
- 设备可处理工艺。
- 区域资源与危险。
- 怪物掉落。
- 技能和装备参数。
- 任务解锁条件。
- Wiki 公开等级。
- 官方工具可计算关系。

当前不要求立刻实现数据系统，但后续设计文档应避免只写自然语言描述而无法转成数据。

## 输入与界面兼容

当前首要输入方式：

- 鼠标。
- 键盘。

需要预留：

- 手柄可选适配。
- Web 触控或移动端简化操作。
- Android 官方工具页面的触控浏览。

原则：

- 核心规则不直接依赖某一种输入设备。
- UI 操作可以有平台差异，但命令结果应一致。
- 高密度面板在桌面优先，小屏需要后续专门布局，不能简单缩放。

## 里程碑兼容复核

每个阶段或里程碑完成后，必须进行一次兼容复核，避免前期实现拖垮后期完善。

复核至少包含：

- Windows 主体验是否仍然优先。
- Web 游戏客户端是否仍只是可选试玩 / 技术验证。
- Android 是否仍只是远期评估。
- WebApp 是否仍聚焦 Wiki 和工具，而不是完整游戏客户端。
- GDScript 与 .NET / C# 边界是否仍清楚。
- 存档、静态数据、任务、地图对象和 UI 是否仍保持可迁移、可复用、可扩展。

详细清单见 `docs/planning/milestone-review-checklist.md`。

## 当前阶段不做

当前不做：

- 正式发布流程。
- Steamworks 接入。
- Android 客户端承诺。
- 完整游戏 Web 客户端承诺。
- 云存档。
- 官方账号。
- 专用服务器。
- 跨平台存档同步。
- 完整 Mod 平台。

但当前需要避免：

- 把存档写成无法迁移。
- 把数据写成无法被 Wiki / 官方工具复用。
- 把 UI 设计成只能桌面悬停才能理解。
- 把单机核心逻辑写死为无法联机扩展。
- 把客户端核心逻辑绑定到会阻碍 Web 试玩或多端评估的技术路线上。

## 当前阶段结论

RadishCatalyst 的平台和兼容方向是：

- Windows 桌面版优先，是核心游戏主平台。
- Web 游戏客户端只作为可选试玩 Demo / 技术验证，不作为 MVP 必须交付。
- Android 只做远期评估和工具端友好预留，不作为首版约束。
- WebApp 重点是 Wiki、配方查询器、工艺链规划器、地图 / 资源工具，不是完整游戏客户端。
- Godot 客户端首版以 GDScript 为基线，.NET / C# 优先用于外部工具、服务端和后续可选模块。
- 首版必须离线单机可玩。
- 本地存档是首版核心能力。
- 数据 ID、结构化数据和存档版本从第一天开始考虑。
- 联机是后续扩展，但单机架构不能堵死 Host、专服和官方服务路径。
- 每个里程碑完成后都要做兼容复核，确认平台、存档、数据、任务和工程边界没有被短期实现破坏。
