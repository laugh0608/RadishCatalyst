# Platform And Compatibility

更新时间：2026-04-25

## 文档目的

这份文档用于在正式初始化 Godot 工程和拆 MVP 功能之前，先明确 RadishCatalyst 的首发平台、发布顺序、离线单机要求、存档原则和长期兼容边界。

它不是构建脚本说明，也不是发行计划。它只固定上位技术方向，避免前期做出会阻碍未来 Web、Android、联机、Wiki 和官方工具的结构选择。

## 平台优先级结论

当前平台优先级为：

1. Windows 桌面版。
2. Web 试玩版或 Demo 页面。
3. Android 远期评估。
4. 其他平台暂不纳入当前决策。

首发产品形态优先考虑 Windows。本项目的主要操作、信息密度、基地规划和 ARPG 战斗都更适合鼠标键盘与桌面屏幕。

Web 更适合传播、试玩和小范围验证，不应在首版阶段反向限制核心体验。

Android 只做方向预留，不作为当前首版目标。

## Windows 桌面版

Windows 是首发客户端优先级最高的平台。

原因：

- Godot 4.x 对 Windows 桌面导出成熟。
- 鼠标键盘适合基地建设、设备面板、地图反查和战斗操作。
- 本地文件存档、日志和调试更直接。
- 长期更接近 Steam 等桌面发行路径。

Windows 版本必须支持：

- 离线单机。
- 本地存档。
- 本地设置。
- 可导入 / 导出或备份存档的结构预留。
- 后续接入房间联机、专用服务器或官方服务时的兼容路径。

## Web 试玩版

Web 的定位是试玩和传播，不是首版完整体验的约束来源。

适合承载：

- 早期 Demo。
- 核心操作验证。
- 小规模战斗或基地片段展示。
- 官网、活动页或社区传播。

Web 版本需要注意：

- 文件系统能力受限。
- 性能、内存和加载包大小更敏感。
- 多线程、网络、文件持久化和浏览器兼容性可能限制完整玩法。
- 复杂本地存档需要封装，不应直接依赖桌面路径。

因此从数据和存档设计上要预留抽象层，但不要因为 Web 限制而牺牲 Windows 主体验。

## Android 远期评估

Android 暂不作为首发目标。

保留它的原因：

- 后续可能适合轻量试玩、辅助工具或简化客户端。
- 官方工具、Wiki 和合成台天然适合移动端访问。
- 如果核心游戏后续支持触控，Android 可成为扩展平台。

当前不承诺：

- 首版 Android 客户端。
- 与桌面完全一致的操作体验。
- 在小屏幕上承载完整基地规划和高密度设备面板。

当前需要预留：

- UI 不要完全依赖极小点击目标。
- 核心数据结构和工具页面适合响应式展示。
- 输入层不要把鼠标键盘逻辑写死进核心规则。

## 离线单机原则

首版必须离线单机可玩。

含义：

- 没有网络也能进入游戏。
- 核心循环不依赖账号、服务器、云存档或在线验证。
- 本地世界存档和玩家档案足以支撑完整进度。
- 官方服务失败时，不影响单机游玩。

这条原则优先级高于联机扩展。

## 本地存档原则

本地存档必须从一开始按长期兼容设计。

建议至少区分：

- 世界存档：地图、基地、设备、区域、任务、污染和世界事件。
- 玩家档案：角色等级、装备、技能、背包、快捷栏、设置和个人进度。
- 全局设置：图像、音频、输入、语言、辅助功能等。

存档设计原则：

- 有明确版本号。
- 可迁移。
- 可校验。
- 可备份。
- 尽量避免直接序列化不可控对象图。
- 核心状态以结构化数据表达，而不是散落在场景节点临时变量中。

存档路径和具体格式可在工程初始化后再定，但当前文档层面必须先确定：本地存档是首版核心能力，不是后续补丁。

## 未来联机兼容

本项目长期联机方向以 `multiplayer-and-save-architecture.md` 为准：

- 离线单机。
- 玩家开房 / 好友联机。
- 专用服务器。
- 后续可选官方账号与云存档。

为了兼容这些形态，当前单机设计也应保持以下原则：

- 单机世界仍按“本地权威”心智运行。
- 核心交互尽量命令化。
- 游戏规则、状态和表现层分离。
- 世界状态和玩家状态边界清楚。
- 不把服务器、账号或云服务写成核心玩法前提。

## 数据 ID 原则

所有可被存档、同步、反查或公开展示的对象，都应有稳定 ID。

至少包括：

- 物品。
- 材料。
- 配方。
- 设备。
- 建筑。
- 区域。
- 资源点。
- 怪物。
- 掉落表。
- 技能。
- 装备。
- 任务。
- Wiki 条目。
- 官方工具查询对象。

稳定 ID 原则：

- 不使用显示名作为唯一标识。
- ID 使用英文、稳定、可读的命名。
- 显示名、描述和本地化文本与 ID 分离。
- ID 一旦进入存档或公开数据，应避免随意改名。
- 必须改名时，需要迁移规则或别名表。

## 结构化数据原则

RadishCatalyst 天然包含大量可查询关系，因此数据不能只写在散文档里。

未来应尽量结构化的内容：

- 物品与分类。
- 配方输入、输出和副产物。
- 设备可处理工艺。
- 区域资源与危险。
- 怪物掉落。
- 技能和装备参数。
- 任务解锁条件。
- Wiki 公开等级。
- 官方工具可计算关系。

当前不要求立刻实现数据系统，但后续设计文档应避免只写自然语言描述而无法转成数据。

## 输入与界面兼容

当前首要输入方式：

- 鼠标。
- 键盘。

需要预留：

- 手柄可选适配。
- Web 触控或移动端简化操作。
- Android 官方工具页面的触控浏览。

原则：

- 核心规则不直接依赖某一种输入设备。
- UI 操作可以有平台差异，但命令结果应一致。
- 高密度面板在桌面优先，小屏需要后续专门布局，不能简单缩放。

## 当前阶段不做

当前不做：

- 正式发布流程。
- Steamworks 接入。
- Android 客户端承诺。
- 云存档。
- 官方账号。
- 专用服务器。
- 跨平台存档同步。
- 完整 Mod 平台。

但当前需要避免：

- 把存档写成无法迁移。
- 把数据写成无法被 Wiki / 官方工具复用。
- 把 UI 设计成只能桌面悬停才能理解。
- 把单机核心逻辑写死为无法联机扩展。

## 当前阶段结论

RadishCatalyst 的平台和兼容方向是：

- Windows 桌面版优先。
- Web 作为试玩和传播渠道预留。
- Android 只做远期评估和工具端友好预留。
- 首版必须离线单机可玩。
- 本地存档是首版核心能力。
- 数据 ID、结构化数据和存档版本从第一天开始考虑。
- 联机是后续扩展，但单机架构不能堵死 Host、专服和官方服务路径。
