# Save Data Model - 原则与分层

返回：[Save Data Model](save-data-model.md)

## 文档目的

这份文档用于在 Godot 工程初始化前，先定义 RadishCatalyst 首版本地存档、稳定 ID 和核心状态边界。

它不是最终文件格式规范，也不是完整数据库设计。它只回答：

- 首版需要保存哪些状态。
- 哪些数据应来自静态配置，哪些应写入存档。
- 世界存档、角色档案、玩家档案和全局设置如何分工。
- 地图对象、可破坏状态、地形、地基、污染、时间天气、设备和背包如何进入存档。
- 如何为未来房间联机、专用服务器、Wiki 和官方工具预留结构。

## 设计原则

首版存档必须遵守以下原则：

- 离线单机完整可玩。
- 本地存档是核心能力，不是后续补丁。
- 所有进入存档、同步、Wiki 或官方工具的数据都必须有稳定 ID。
- 静态配置和运行时状态分离。
- 不直接序列化 Godot 场景节点或不可控对象图。
- 世界状态、角色状态、玩家设置和全局配置边界清楚。
- 存档有版本号，并为迁移规则预留位置。
- 单机也按“本地权威世界”心智设计，避免未来联机重构。

## 存档分层

首版建议分为四类文件或逻辑块。

### 世界存档

代表“这个世界本身”。

保存：

- 世界 ID。
- 世界名称。
- 存档版本。
- 创建时间和最近保存时间。
- 当前地图、区域和探索状态。
- 基地建筑、设备、管线、储存和库存。
- 地形状态、地基、平整和清理结果。
- 可破坏对象的当前状态。
- 污染状态。
- 时间与天气状态。
- 世界任务、区域解锁和主线进度。
- 世界事件和触发器。

### 角色档案

代表“当前玩家使用的角色”。

保存：

- 角色 ID。
- 角色名。
- 外观或原型。
- 所属或最近进入的世界 ID。
- 位置和朝向。
- 生命、防护、能量或体力。
- 等级、经验和轻量成长进度。
- 装备、模块和快捷栏。
- 背包内容。
- 已知配方、扫描信息或个人发现。
- 角色任务进度。

首版可以让角色与世界强绑定或弱绑定，但必须明确：世界进度、基地资源、区域解锁和主线推进默认属于世界存档。

### 玩家档案

代表“这个玩家自己的本地资料”。

保存：

- 玩家本地 ID。
- 默认角色列表。
- 本地成就或统计。
- 可选的新手提示状态。
- 最近打开的世界。

首版不需要账号、好友、云存档或服务器列表。

### 全局设置

代表“这个客户端的设置”。

保存：

- 图像设置。
- 音频设置。
- 输入映射。
- 语言。
- 字号、色弱辅助、震动或闪烁强度等辅助功能。
- Web 或桌面平台差异设置。

全局设置不应写进单个世界存档，避免多个世界互相污染。

## 静态配置与运行时状态

### 静态配置

静态配置由游戏数据定义，不应重复完整写入存档。

包括：

- 物品定义。
- 资源定义。
- 配方定义。
- 设备定义。
- 建筑定义。
- 敌人定义。
- 掉落表。
- 区域模板。
- 天气类型。
- 污染类型。
- 装备、模块和技能定义。
- UI 文本和本地化。

存档中只引用这些配置的稳定 ID。

### 运行时状态

运行时状态代表玩家实际改变过的世界。

包括：

- 某个设备是否已建造。
- 某个资源点是否已采集。
- 某个障碍是否已破坏。
- 某块地是否已平整或铺设地基。
- 某片区域污染值是多少。
- 当前天气是什么。
- 当前库存有多少物品。
- 当前任务是否完成。

运行时状态必须可保存、可加载、可迁移。

## 稳定 ID 规则

首版至少需要三类 ID。

### 定义 ID

用于静态配置。

示例：

- `item.crystal_ore`
- `fluid.polluted_slurry`
- `recipe.crystal_catalyst_basic`
- `building.basic_reactor`
- `equipment.filter_module_t1`
- `enemy.polluted_larva`
- `weather.acid_mist`
- `region.crystal_vein_field`

规则：

- 使用英文。
- 稳定、可读、可分类。
- 不使用显示名。
- 一旦进入存档或公开数据，不随意改名。

### 实例 ID

用于世界中已经生成或建造的具体对象。

示例：

- `world_001:building:000023`
- `world_001:container:000004`
- `world_001:resource_node:000180`
- `world_001:enemy:000392`

规则：

- 在同一个世界内唯一。
- 可由世界 ID、对象类型和递增序号组成。
- 不依赖场景节点路径。
- 不依赖显示名。

### 存档槽 ID

用于区分本地世界、角色和备份。

示例：

- `world_20260426_001`
- `character_20260426_001`
- `backup_world_20260426_001_0003`

规则：

- 对玩家可隐藏。
- 支持重命名显示名。
- 支持备份和导入导出。
