## 7. 存档归属模型

建议明确区分三类存档或档案：

### 7.1 世界存档

用于保存：

- 地图进度
- 建筑布局
- 设备状态
- 管网状态
- 储罐内容
- 电网状态
- 区域解锁
- 主线推进
- 世界任务状态
- 基地库存

它代表的是“这个世界本身”。

### 7.2 玩家档案

用于保存：

- 账号显示名或头像设置
- 输入设置
- 辅助功能设置
- 已知服务器、好友和账号信息
- 成就或全局解锁

它代表的是“这个玩家自己”。

玩家档案不应直接等同于某一个角色。一个玩家可以拥有多个角色。

### 7.3 角色档案

用于保存：

- 角色 ID
- 角色名和外观
- 初始背景或角色原型
- 初始能力
- 初始天赋
- 等级
- 技能
- 装备
- 快捷栏
- 个人背包
- 个人任务进度
- 与某个世界的绑定关系

它代表的是“这个玩家当前使用的具体角色”。

### 7.4 单机模式下的关系

单机时，世界存档和玩家档案可以一起保存在本地。

同一个玩家可以在不同本地世界中选择不同角色。不同角色可以拥有不同初始能力和天赋，用于支持不同 Build、不同开局体验和不同协作分工。

### 7.5 开房模式下的关系

开房时：

- 世界存档归房主
- 其他玩家带入自己的角色档案，选择一个角色加入，或使用房间规则指定的临时角色

建议优先考虑三种模式：

1. 角色绑定世界  
进入某个世界时，角色成长跟随该世界推进

2. 角色弱绑定世界
角色可以由玩家选择进入多个世界，但等级、装备、背包或关键进度根据世界规则限制

3. 临时房间角色
加入某个世界时按房间规则生成临时角色，离开后只保存有限个人进度或不保存

从开发复杂度和平衡看，首版建议优先：

**角色与世界弱绑定，但跨世界资源、装备、配方和任务推进携带要严格限制。**

推荐规则：

- 同一个玩家可以创建多个角色。
- 不同存档可以选择不同角色。
- 同一角色进入不同世界时，可以保留外观、基础天赋和部分个人成长。
- 资源、装备、基地解锁、区域进度和主线推进默认跟随世界存档。
- 房主或服务器应能决定是否允许外部角色加入。
- 如果允许外部角色加入，应有等级、装备、背包或材料携带限制，避免破坏世界进度。

### 7.6 初始能力与天赋

不同角色可以有不同初始能力和天赋，但它们应服务协作和 Build 倾向，而不是变成不可逆职业锁死。

推荐初始倾向：

- 工程倾向：建造、修复、部署装置或设备排障更强。
- 外勤倾向：移动、采集、扫描或背包效率更强。
- 战斗倾向：武器操控、输出、控场或弱点打击更强。
- 生存倾向：环境抗性、治疗、撤离或补给效率更强。

设计边界：

- 初始天赋不能替代基地生产和装备成长。
- 不同角色应能通过装备、模块和技能形成混合 Build。
- 首版不做复杂职业树，优先做少量清晰初始差异。
- 联机协作可以鼓励分工，但单机角色必须也能完整通关首版闭环。

### 7.7 本地存档阶段策略

当前原型已经具备默认槽位单文件存档、三槽位原型 UI 入口、最近 3 份 `.bak.1` / `.bak.2` / `.bak.3` 备份轮转、坏主档备份读取恢复、旧原型存档迁移、首批内容一致性校验、首批实例与跨块关系校验、地图对象 / 敌人 / 建筑运行时来源复验、配方与区域解锁来源追溯、任务链来源复验、任务目标类型复验、任务目标进度来源与上限复验，以及已完成任务目标满足度校验。后续不要直接跳到复杂云存档或正式多角色 UI，应按以下阶段推进。

#### 阶段 0：第一切片原型

当前阶段保持简单：

- 存档目录使用 Godot `user://saves/slots/slot_01/` 作为当前默认槽位。
- 主存档文件为 `slot_01/slice_01_autosave.json`。
- 覆盖保存前轮转 `slot_01/slice_01_autosave.bak.1.json`、`slot_01/slice_01_autosave.bak.2.json` 和 `slot_01/slice_01_autosave.bak.3.json`。
- 存档内容同时包含 `world` 与 `character`，方便原型闭环和运行时复验。
- 读取时按主存档、`bak.1`、`bak.2`、`bak.3` 依次寻找可用档；全部失败时必须保留当前运行状态，不用坏档覆盖内存中的可玩状态。
- 原型 HUD 已提供 `slot_01`、`slot_02` 和 `slot_03` 三个按钮槽位，显示空槽、可读取、备份可恢复或不可读取状态；`K` / `L` 仍保留为默认 `slot_01` 的调试快捷键。
- 如果默认槽位完全缺失，会尝试读取旧路径 `user://saves/slice_01_autosave.json` 及旧备份，校验成功后写入 `slot_01`，并保留旧文件作为导入来源。

这个阶段只服务第一切片验证，不代表正式存档布局。

#### 阶段 1：首版本地单槽

进入首版前，应把原型单文件升级为“一个世界槽位 + 自动存档 + 备份轮转”：

```text
user://saves/
  worlds/
    world_<stable_id>/
      metadata.json
      autosave.json
      backups/
        autosave.bak.1.json
        autosave.bak.2.json
        autosave.bak.3.json
```

规则：

- `metadata.json` 保存世界显示名、创建时间、更新时间、版本、游玩时长和最近区域等摘要。
- `autosave.json` 是默认进入世界时读取的主文件。
- `backups/` 中保留最近 3 份可读取备份。
- 备份轮转顺序为：写入新主存档前，将 `bak.2` 移为 `bak.3`，`bak.1` 移为 `bak.2`，当前 `autosave.json` 复制为 `bak.1`。
- 如果备份轮转失败，不允许覆盖当前主存档。

#### 阶段 2：首版多槽位

多槽位不应改变“一个世界对应一个存档”的根模型，只是在 `worlds/` 下增加多个世界目录：

```text
user://saves/
  profile/
    local_profile.json
  characters/
    character_<stable_id>.json
  worlds/
    world_<stable_id>/
      metadata.json
      autosave.json
      manual_001.json
      backups/
```

规则：

- 世界进度、基地、区域、任务、建筑、设备和世界库存归属 `worlds/world_<stable_id>/`。
- 角色外观、装备、快捷栏和个人背包先随世界存档保存；等角色档案边界稳定后，再迁出到 `characters/`。
- `local_profile.json` 只保存本地设置、最近进入世界、显示偏好和非关键个人配置。
- 手动存档可以作为同一世界下的额外快照，但不能变成跨世界资源转移入口。

#### 阶段 3：迁移与恢复

正式进入多槽位后，仍应保留从当前 `slot_01` 和旧原型路径导入世界目录的迁移路径：

1. 如果发现 `slice_01_autosave.json`，先按旧格式读取并校验。
2. 读取成功后生成新的 `world_<stable_id>/metadata.json` 和 `autosave.json`。
3. 原旧档移动到 `legacy/` 或保留为只读导入源，不在原路径继续覆盖。
4. 迁移失败时保留旧档，并显示明确失败原因。

坏档恢复优先级：

1. 尝试读取主存档。
2. 主存档缺失或校验失败时，尝试读取 `backups/autosave.bak.1.json`。
3. 继续失败则尝试 `bak.2`、`bak.3`。
4. 所有备份失败时，保留坏档文件，创建安全默认世界，并提示玩家原存档不可读取。

恢复成功后，不应立刻覆盖原坏档；应等玩家再次主动保存或自动保存触发时再写入新的主存档。

### 7.8 联机与云存档边界

首版本地存档不依赖账号、网络或官方服务。

未来接入联机和云存档时，边界应保持：

- 房主开房时，世界存档仍归房主本地或专服。
- 专用服务器使用同一套世界状态结构，但保存位置从 `user://saves/` 迁到服务器数据目录。
- 官方云存档优先同步玩家档案、角色档案和世界备份元数据，不优先实时同步完整世界。
- 云同步必须以本地可读存档为基础，不能让离线单机变成在线依赖。
- 多人世界的冲突合并不在首版处理；Host / 专服是唯一权威写入者。
