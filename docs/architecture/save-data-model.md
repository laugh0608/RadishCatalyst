# Save Data Model

更新时间：2026-04-29

## 文档目的

这份文档用于在 Godot 工程初始化前，先定义 RadishCatalyst 首版本地存档、稳定 ID 和核心状态边界。

它不是最终文件格式规范，也不是完整数据库设计。它只回答：

- 首版需要保存哪些状态。
- 哪些数据应来自静态配置，哪些应写入存档。
- 世界存档、角色档案、玩家档案和全局设置如何分工。
- 地图对象、可破坏状态、地形、地基、污染、时间天气、设备和背包如何进入存档。
- 如何为未来房间联机、专用服务器、Wiki 和官方工具预留结构。

## 设计原则

首版存档必须遵守以下原则：

- 离线单机完整可玩。
- 本地存档是核心能力，不是后续补丁。
- 所有进入存档、同步、Wiki 或官方工具的数据都必须有稳定 ID。
- 静态配置和运行时状态分离。
- 不直接序列化 Godot 场景节点或不可控对象图。
- 世界状态、角色状态、玩家设置和全局配置边界清楚。
- 存档有版本号，并为迁移规则预留位置。
- 单机也按“本地权威世界”心智设计，避免未来联机重构。

## 存档分层

首版建议分为四类文件或逻辑块。

### 世界存档

代表“这个世界本身”。

保存：

- 世界 ID。
- 世界名称。
- 存档版本。
- 创建时间和最近保存时间。
- 当前地图、区域和探索状态。
- 基地建筑、设备、管线、储存和库存。
- 地形状态、地基、平整和清理结果。
- 可破坏对象的当前状态。
- 污染状态。
- 时间与天气状态。
- 世界任务、区域解锁和主线进度。
- 世界事件和触发器。

### 角色档案

代表“当前玩家使用的角色”。

保存：

- 角色 ID。
- 角色名。
- 外观或原型。
- 所属或最近进入的世界 ID。
- 位置和朝向。
- 生命、防护、能量或体力。
- 等级、经验和轻量成长进度。
- 装备、模块和快捷栏。
- 背包内容。
- 已知配方、扫描信息或个人发现。
- 角色任务进度。

首版可以让角色与世界强绑定或弱绑定，但必须明确：世界进度、基地资源、区域解锁和主线推进默认属于世界存档。

### 玩家档案

代表“这个玩家自己的本地资料”。

保存：

- 玩家本地 ID。
- 默认角色列表。
- 本地成就或统计。
- 可选的新手提示状态。
- 最近打开的世界。

首版不需要账号、好友、云存档或服务器列表。

### 全局设置

代表“这个客户端的设置”。

保存：

- 图像设置。
- 音频设置。
- 输入映射。
- 语言。
- 字号、色弱辅助、震动或闪烁强度等辅助功能。
- Web 或桌面平台差异设置。

全局设置不应写进单个世界存档，避免多个世界互相污染。

## 静态配置与运行时状态

### 静态配置

静态配置由游戏数据定义，不应重复完整写入存档。

包括：

- 物品定义。
- 资源定义。
- 配方定义。
- 设备定义。
- 建筑定义。
- 敌人定义。
- 掉落表。
- 区域模板。
- 天气类型。
- 污染类型。
- 装备、模块和技能定义。
- UI 文本和本地化。

存档中只引用这些配置的稳定 ID。

### 运行时状态

运行时状态代表玩家实际改变过的世界。

包括：

- 某个设备是否已建造。
- 某个资源点是否已采集。
- 某个障碍是否已破坏。
- 某块地是否已平整或铺设地基。
- 某片区域污染值是多少。
- 当前天气是什么。
- 当前库存有多少物品。
- 当前任务是否完成。

运行时状态必须可保存、可加载、可迁移。

## 稳定 ID 规则

首版至少需要三类 ID。

### 定义 ID

用于静态配置。

示例：

- `item.crystal_ore`
- `fluid.polluted_slurry`
- `recipe.crystal_catalyst_basic`
- `building.basic_reactor`
- `equipment.filter_module_t1`
- `enemy.polluted_larva`
- `weather.acid_mist`
- `region.crystal_vein_field`

规则：

- 使用英文。
- 稳定、可读、可分类。
- 不使用显示名。
- 一旦进入存档或公开数据，不随意改名。

### 实例 ID

用于世界中已经生成或建造的具体对象。

示例：

- `world_001:building:000023`
- `world_001:container:000004`
- `world_001:resource_node:000180`
- `world_001:enemy:000392`

规则：

- 在同一个世界内唯一。
- 可由世界 ID、对象类型和递增序号组成。
- 不依赖场景节点路径。
- 不依赖显示名。

### 存档槽 ID

用于区分本地世界、角色和备份。

示例：

- `world_20260426_001`
- `character_20260426_001`
- `backup_world_20260426_001_0003`

规则：

- 对玩家可隐藏。
- 支持重命名显示名。
- 支持备份和导入导出。

## 世界存档结构建议

首版世界存档可以按逻辑块组织：

```text
world
metadata
regions
map_cells
map_objects
base
inventories
pollution
time_weather
quests
events
```

### metadata

保存：

- 存档版本。
- 游戏版本。
- 世界 ID。
- 世界显示名。
- 创建时间。
- 最近保存时间。
- 最近使用角色 ID。

### regions

保存区域级状态：

- 区域 ID。
- 是否已发现。
- 是否已解锁。
- 探索比例或关键节点进度。
- 危险等级修正。
- 敌对生态活跃度。
- 区域目标状态。

### map_cells

保存地块级状态。

首版建议使用离散地块或网格心智，即使实际 Godot 表现不是严格网格，也应有可序列化的地块数据。

保存：

- 坐标。
- 高度等级。
- 地表类型。
- 是否已清理。
- 是否已平整。
- 是否有地板或地基。
- 地基等级。
- 可建造性覆盖。
- 污染值或污染类型引用。
- 是否已探索。

不建议保存：

- 每帧视觉效果。
- 可由地块状态推导出的装饰细节。
- 临时粒子、脚印或短期战斗痕迹。

### map_objects

保存地图对象实例。

对象包括：

- 资源节点。
- 样本节点。
- 可破坏障碍。
- 不可破坏但需要记录发现状态的遗迹节点。
- 敌人巢穴。
- 掉落物。
- 任务交互点。

保存：

- 实例 ID。
- 定义 ID。
- 位置。
- 朝向或形态。
- 当前状态，例如完整、受损、已破坏、已采集、已采样、已净化。
- 当前耐久或剩余资源量。
- 是否已被玩家发现。
- 关联任务或区域事件。

原则：

- 不可破坏对象通常不需要保存“耐久”，但可能需要保存是否发现、是否激活、是否解锁。
- 可破坏对象必须保存当前状态，避免加载后复原。
- 可采集对象应保存剩余量或已采集状态。

### base

保存基地和工业系统状态。

包括：

- 基地核心。
- 建筑和设备实例。
- 管线、泵、储罐和连接关系。
- 电力或能源网络。
- 设备运行状态。
- 设备配方选择。
- 设备输入、输出和内部缓冲。
- 设备故障、堵塞、过热或污染超标状态。

设备实例保存：

- 实例 ID。
- 定义 ID。
- 位置和占地。
- 朝向。
- 所在地基或地块引用。
- 当前配方 ID。
- 运行状态。
- 处理进度。
- 内部库存。
- 能源需求和当前供给状态。
- 异常状态。

### inventories

保存库存状态。

库存类型：

- 基地总库存。
- 设备内部缓存。
- 箱体。
- 储罐。
- 玩家背包。
- 地面掉落。

物品库存保存：

- 物品 ID。
- 数量。
- 可选品质或耐久。

流体库存保存：

- 流体 ID。
- 数量。
- 可选纯度、温度、污染度或稳定性。

首版可以先不做复杂品质、温度和压力，但字段位置应预留。

### pollution

保存污染状态。

建议分层：

- 区域污染总览。
- 地块污染值。
- 污染源点状态。
- 污染处理设施状态。
- 污染事件或生态反应计数。

污染字段：

- 污染类型。
- 当前数值。
- 趋势或最近变化。
- 来源引用。
- 是否已治理。
- 对敌人、建造或防护消耗的影响标记。

### time_weather

保存时间与天气状态。

首版至少预留：

- 世界时间。
- 日期或周期计数。
- 当前时间段，例如白天、夜晚。
- 当前天气 ID。
- 天气剩余时间。
- 下一个天气候选或随机种子。
- 区域天气覆盖。

原则：

- 天气必须可保存，避免加载后随机跳变。
- 天气影响应可由天气 ID 和区域状态推导。
- 不做复杂气候模拟时，也应保留扩展字段。

### quests

保存任务和解锁。

包括：

- 主线阶段。
- 区域目标。
- 已完成教程提示。
- 已解锁配方。
- 已解锁设备。
- 已解锁区域入口。
- 已分析样本。

### events

保存世界事件。

包括：

- 已触发但未完成的事件。
- 已完成事件。
- 敌袭或污染反应计时。
- 秘境入口状态。
- 事故记录或当前事故。

首版可以简单，但不应把事件状态散落在场景节点里。

## 角色档案结构建议

角色档案可以按逻辑块组织：

```text
character
metadata
status
progression
equipment
inventory
known_data
```

### status

保存：

- 当前世界 ID。
- 当前区域 ID。
- 位置。
- 朝向。
- 生命。
- 防护完整度。
- 能量或体力。
- 当前环境状态，例如中毒、腐蚀、低温、污染暴露。
- 复活点或撤离点。

### progression

保存：

- 等级。
- 经验。
- 已解锁基础能力。
- 外勤专精或轻量天赋。
- 区域适应进度。

### equipment

保存：

- 武器槽。
- 防护服槽。
- 模块槽。
- 快捷栏。
- 当前弹药或消耗品绑定。

装备实例可以先用定义 ID + 耐久 + 模块组合表达。首版不建议引入复杂随机词条实例。

### inventory

保存角色背包：

- 物品 ID。
- 数量。
- 可选耐久或状态。

背包容量、负重和任务物品规则应可由角色状态和配置推导。

### known_data

保存个人发现：

- 已扫描资源。
- 已识别敌人。
- 已见过天气或污染类型。
- 已发现区域提示。

这些信息未来可服务 Wiki 解锁、提示和官方工具公开等级。

## 命令与存档关系

为了兼容未来联机，关键交互应尽量命令化。

首版建议至少定义这些命令心智：

- `MoveCharacter`
- `Attack`
- `UseTool`
- `CollectResource`
- `TakeSample`
- `DamageMapObject`
- `ClearTile`
- `LevelTile`
- `PlaceFoundation`
- `BuildStructure`
- `RemoveStructure`
- `SetRecipe`
- `TransferItem`
- `UseConsumable`
- `UnlockRegion`

存档保存命令执行后的状态，不需要保存完整命令日志。

但调试和未来回放可以可选记录近期命令，用于复现错误。

## 文件格式建议

工程初始化前暂不强制具体格式。

建议原则：

- 首版可使用 JSON 或 Godot 可稳定读写的文本 / 二进制资源格式。
- 开发早期优先可读、可 diff、可排查。
- 世界存档较大后，可以拆分区域块或使用压缩。
- 不把存档和静态配置混在一个文件。
- 不在存档中保存本地化显示文本。

推荐早期结构：

```text
saves/
  worlds/
    world_20260426_001/
      world.json
      regions.json
      map_cells.json
      map_objects.json
      base.json
      inventories.json
      pollution.json
      time_weather.json
      quests.json
  characters/
    character_20260426_001.json
  player_profile.json
  settings.json
```

后续如果单文件更方便，也可以合并，但逻辑边界不应消失。

## 当前原型实现

截至 2026-04-29，Godot 客户端已接入最小 `SaveService`：

- 存档脚本位于 `client/scripts/save/save_service.gd`。
- 当前使用 `user://saves/slice_01_autosave.json` 作为原型单文件存档。
- `K` 保存当前世界和角色状态，`L` 读取并恢复。
- 当前保存 `save_schema_version`、`game_version`、`created_at`、`updated_at`、`WorldState.to_dict()` 和 `CharacterState.to_dict()`，覆盖任务、区域解锁、地图对象、敌人、建筑、污染、天气、生命、防护、位置、装备、快捷栏和背包。
- 当前读取会先校验 `save_schema_version`、`world` 和 `character` 是否存在且类型正确；文件不存在、JSON 解析失败、版本不兼容、关键块缺失或关键块类型错误时，会返回清楚的中文失败提示，并避免替换当前运行中的世界和角色状态。
- 当前 `WorldState`、`CharacterState`、`InventoryState` 和 `QuestState` 的 `from_dict()` 会对缺失或类型不符的非关键嵌套字段保留默认值，服务旧原型存档和轻度坏档兜底。
- 当前新增 `scripts/check-client-save.ps1` 和 `client/scripts/checks/save_service_check.gd`，用隔离的 Godot 用户目录复验文件缺失、坏 JSON、版本错误、关键块缺失 / 类型错误、非关键字段默认值兜底，以及默认状态保存后读取。
- 该实现用于验证第一可玩切片状态可落盘，不代表最终存档槽、备份、迁移或导入导出设计。

## 迁移与校验

每个存档必须有：

- `save_schema_version`
- `game_version`
- `created_at`
- `updated_at`

加载时应检查：

- 版本是否支持。
- 必需字段是否存在。
- 引用的定义 ID 是否仍存在。
- 实例 ID 是否重复。
- 库存数量是否有效。
- 坐标、区域和地块引用是否有效。

未来修改 ID 时，应使用：

- ID 别名表。
- 迁移脚本。
- 默认值填充。
- 失败时清楚提示，而不是静默损坏。

当前原型已完成最小读取校验和运行时复验入口：`save_schema_version`、`world` 和 `character` 为关键块，必须存在且类型正确；其余嵌套字段暂按默认值兜底。后续进入多槽位、备份或正式迁移前，应继续补充静态定义 ID、库存数量、坐标和实例 ID 的复验。

## 首版必须保存的最小集合

首版 MVP 至少必须保存：

- 世界元数据。
- 已探索区域。
- 区域解锁。
- 地块清理、平整、地基和污染状态。
- 可采集、可采样、可破坏对象状态。
- 基地设备、库存、配方和处理进度。
- 角色位置、生命、防护、装备、背包和快捷栏。
- 已解锁配方、模块、药剂或区域通行能力。
- 当前时间段和天气状态。

如果某个状态会影响玩家进度、资源、区域、基地或角色能力，就必须进入存档。

## 当前不做

首版存档暂不做：

- 云存档。
- 账号绑定。
- 跨设备同步。
- 专用服务器持久化。
- 多人权限状态。
- 完整命令日志回放。
- 复杂随机地图重建。
- 大规模分块流式存档优化。

但当前必须避免：

- 用场景节点路径当长期 ID。
- 用显示名当 ID。
- 把库存、设备、污染、任务状态散落在 UI 节点里。
- 让天气、地形、可破坏对象只存在于临时场景状态。
- 存档里保存大量可由配置推导的数据。

## 与后续文档的关系

Godot 工程初始化后，应把本文件拆成具体实现：

- 静态数据目录和 ID 命名规则。
- SaveService 或等价存档服务。
- WorldState、CharacterState、InventoryState 等状态对象。
- 地图对象实例注册和实例 ID 生成。
- 存档加载、保存、迁移和校验流程。

如果后续新增联机、专服或官方云存档，应优先复用本文件的世界 / 角色 / 玩家 / 设置分层，而不是重写一套状态模型。

## 当前阶段结论

RadishCatalyst 的首版存档重点不是格式，而是边界。

当前最重要的结论是：

**静态配置用稳定定义 ID，运行时状态用世界内实例 ID；世界保存地图、基地、污染、时间天气和任务，角色保存生命、装备、背包和成长，设置独立保存。**

只要这条边界稳定，后续 Godot 原型、玩家 Wiki、官方工具和联机演进都会更容易接上。
