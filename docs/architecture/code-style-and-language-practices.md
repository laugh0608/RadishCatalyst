# Code Style And Language Practices

更新时间：2026-05-06

## 目标

本文定义 RadishCatalyst 的代码语言实践约束。重点不是统一所有格式细节，而是让代码贴近对应语言和框架的良好实践，保持直白、可维护、可验证。

## 通用原则

- 命名应表达领域意图、输入输出和行为结果，避免只有作者当时理解的缩写或隐喻。
- 逻辑优先写清楚，再考虑抽象；抽象必须能降低复杂度、减少真实重复、隔离稳定边界或表达明确领域概念。
- 优先使用标准库、引擎 API、结构化数据和明确类型表达意图。
- 不用脆弱的字符串拼接、隐式约定或注释掩盖本可以写清楚的实现。
- 注释只解释必要背景、非显然约束、兼容原因或设计权衡，不重复代码字面含义。

## GDScript / Godot

- Godot 客户端默认优先使用 GDScript，遵循 Godot 4.x 的节点、资源、信号、场景和生命周期习惯。
- 能用节点引用、导出属性、资源、信号或引擎 API 清晰表达时，不自建绕远的全局转发表或字符串协议。
- UI 脚本应区分控件胶水、展示文本规则和状态来源；拆分要服务于可读性和验证，不为了“拆分”制造新层级。
- 数据驱动内容应优先走已有静态数据 schema 和检查脚本，不把数据规则散落进场景脚本。
- 脚本接近 1000 行时，后续新增实现优先按职责拆分；硬上限为 1500 行。

## PowerShell / Shell

- 脚本应使用清晰参数、明确错误处理和可读的管道表达验证逻辑。
- 不用复杂字符串拼接替代对象、表格、JSON 或已有结构化能力。
- 检查脚本输出应便于定位失败原因，避免只返回笼统失败。
- 跨平台入口存在时，PowerShell 与 Shell 版本应保持检查口径一致。

## 数据与配置

- JSON、CSV、Markdown 等文件应保持稳定 schema、清晰字段命名和可自动检查的约束。
- 新增字段应优先说明来源、用途和运行时影响；不要为未来假设预留大量未使用字段。
- 数据校验应尽量进入现有 `scripts/check-client-*.ps1` 或 Godot 检查脚本，避免只靠人工记忆。

## 禁止事项

- 禁止不明意义的方法名、类名或文件名。
- 禁止晦涩难懂的抽象封装、空泛 `Manager` / `Helper` / `Service` 包装和多层无收益转发。
- 禁止为了显得“高级”提前引入泛化框架、插件系统、事件总线或复杂继承层级。
- 禁止用大量兜底逻辑掩盖根因；应优先修正数据、状态、接口或生命周期边界。

## 验证入口

客户端相关改动优先执行：

```powershell
pwsh ./scripts/check-client.ps1
```

提交前执行：

```powershell
pwsh ./scripts/check-text-files.ps1
git diff --check
```

涉及具体脚本、数据、任务、场景或存档时，应按改动范围补充对应单项验证，并把重要阶段性验证记录到本周周志。
