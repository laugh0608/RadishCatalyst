# Development Retest Baselines

更新时间：2026-05-17

## 文档目的

这份文档定义首小时原型当前使用的“开发复测基线”。

它回答三件事：

- 为什么不应再把“每次人工复测都从空档重打”当默认。
- 当前有哪些标准阶段基线，分别适合复测哪一段。
- 开发时应该怎样在“局部基线复测”和“全链路空档复测”之间做取舍。

## 设计目标

开发复测基线的目标不是替代空档复测，而是把空档复测从“每次都做”降级为“在真正需要时做”。

当前设计遵守以下原则：

- 基线由代码生成，不手工维护长期 JSON 快照。
- 基线必须同时覆盖任务状态、区域解锁、场景完成态、敌人状态、已建结构和关键库存。
- 基线必须能直接载入当前运行时，不要求先跑外部脚本。
- 载入基线后，若需要长期保留，仍可保存到普通槽位。
- 新内容扩展时，优先补对应基线，而不是让人工复测链无限变长。

当前运行时实现位于：

- `client/scripts/save/development_baseline_catalog.gd`
- `client/scripts/save/development_baseline_builder.gd`
- `client/scripts/ui/hud_development_baseline_presenter.gd`

## 当前基线

| 基线 | 阶段状态 | 默认用途 |
| --- | --- | --- |
| `S0` | 新档，恢复前哨前 | 全链路冷启动、早期任务理解、共享系统大回归 |
| `S1` | 处理点扩建完成，准备进入污染边界 | 过滤模块启用、污染边界、药剂和污染战斗 |
| `S2` | 外圈中继确认完成，准备回收外圈回波匣 | 外圈回波、基地解析和第二闭环入口 |
| `S3` | 深段门禁写入完成，准备回收相位纤丝 | 深段入口、过滤器精炼、覆写栓链路 |
| `S4` | 深段样块回收完成，准备基地解析第二轮路线 | 深段样块、阵列点亮、追袭体和相位导管 |
| `S5` | 前线回传锚点和基地相位回投台都已上线 | 前线回传、基地回投、裂相脊两点共振读数和碎屑回收 |
| `S6` | 内层故障轨迹已回收，准备解析相位井锁链 | 内层故障轨迹、故障脉冲读数、残渣回收、井锁整备和相位井定位器 |
| `S7` | 相位井定位器已带回，准备推进内层相位井 | 定位器解析、井涌泄压、井涌碎屑、探针整备和内层相位井 |
| `S8` | 相位井芯样本已带回，准备推进井底裂口 | 井芯样本解析、井底余烬壳清理、井壁余烬、井底穿钉和井底裂口 |
| `S9` | 相位井心核已带回，准备推进井心室断面 | 相位井心核解析、两处分流读数、心棘残片、相位井抑振骨、井心分流栓和井心室断面 |
| `S10` | 相位井纺核已带回，准备推进井纺室断面 | 相位井纺核解析、两处张力绕轮、纬束残团、相位井张力肋、井纺梭栓和井纺室断面 |
| `S11` | 相位井织核已带回，准备推进井纹架断面 | 相位井织核解析、边缕残条、相位井纹架肋、井纹架键栓和井纹架断面 |
| `S12` | 相位井结核已带回，准备推进井系桥断面 | 相位井结核解析、两端结点、系索残股、相位井系固肋、井系定桩和井系桥断面 |
| `S13` | 相位井锚核已带回，准备推进锚场回稳 | 相位井锚核解析、锚核落尘、稳场滤囊、井系校锚桩、压力钉、守脉体、锚场回稳和相位井余响片 |
| `S14` | 锚场回稳已完成，相位井余响片已带回 | 锚场完成态、局部稳定窗口收益、余响片解析和下一包入口判断 |
| `S15` | 稳窗读数已解析，准备返回现场校准稳窗相位序 | 稳窗读数完成态、现场校准目标入口、锚场回访回充和错序反馈 |
| `S16` | 稳窗相位序已校准，准备回基地确认前线行动 | 稳窗校准完成态、前线行动台入口、前线容错收益和下一循环入口 |
| `S17` | 前线行动回报已归档 | 基地确认、稳窗回波探点读取、回基地解析、补给反馈和最短基地-前线-基地闭环 |

这些基线对应的是“标准复测起点”，不是玩家正式存档位。

当前口径补充：

- `S4` 应保留第二轮深段链继续推进所需的基础零件与污染浆液，避免为了验证阵列线 / 回传线被迫回头补前置材料。
- `S5` 若载入到基地，HUD / 地图应直接把玩家指向相位回投台，并在回到裂相脊后要求先写入两处共振读数，再处理猎手和碎屑回收。
- `S5` 到 `S13` 的读数点应显示当前进度和读数完成后打开的回收线；井底余烬壳、井纹架侧路和锚场压力钉应使用各自专用清障提示，不回落到通用地块清理文案。
- 晚期基线载入后，生命 / 防护上限必须按已完成深段里程碑自动同步，不再固定停在 `100 / 100`。
- `S6` 应保留前线回传锚点在线、基地回投链可直接使用，并让玩家在基地起跑时立刻进入“故障轨迹解析 -> 故障脉冲读数 -> 相位井锁”的后续闭环。
- `S6` 之后应能看到相位中继锚点完成态已解锁 `回收基础零件`；若后段配方缺基础零件且背包有污染浆液，基础反应炉推荐应优先切到该回收配方。
- `S7` 应保留相位井定位器已带回且基地起跑可直接解析的状态，让玩家立刻进入“定位器解析 -> 井涌泄压 -> 内层相位井”的后续闭环。
- `S8` 应保留相位井芯样本已带回且基地起跑可直接解析的状态，让玩家立刻进入“井芯样本解析 -> 井底清障 -> 井底整备 -> 井底裂口”的后续闭环。
- `S9` 应保留相位井心核已带回且基地起跑可直接解析的状态，让玩家立刻进入“相位井心核解析 -> 井心整备 -> 井心室断面”的后续闭环。
- `S10` 应保留相位井纺核已带回且基地起跑可直接解析的状态，让玩家立刻进入“相位井纺核解析 -> 井纺整备 -> 井纺室断面”的后续闭环。
- `S11` 应保留相位井织核已带回且基地起跑可直接解析的状态，让玩家立刻进入“相位井织核解析 -> 井纹架整备 -> 井纹架断面”的后续闭环。
- `S12` 应保留相位井结核已带回且基地起跑可直接解析的状态，让玩家立刻进入“相位井结核解析 -> 井系整备 -> 井系桥断面”的后续闭环。
- `S13` 应保留相位井锚核已带回且基地起跑可直接解析的状态，让玩家立刻进入“相位井锚核解析 -> 锚场整备 -> 锚场回稳”的非同构后续闭环。
- `S14` 应保留锚场回稳已完成、相位井余响片已带回、稳定窗口已生成的状态，让下一轮可以直接回归完成态收益和余响片解析入口，而不是重复跑锚核解析。
- `S15` 应保留相位井稳窗读数已解析、锚场回稳窗已可回访的状态，并激活三处稳窗节点现场校准任务；它是第一版现场校准目标的复测入口。
- `S16` 应保留西侧、中央和东侧三处稳窗校准点已按顺序写入的状态，并激活前线行动台确认任务，让下一轮可以直接验证 `S16` 后的基地入口。
- `S17` 应保留前线行动台已确认、稳窗回波探点已读取、前线行动回报已解析和短行动补给已整理的状态，让下一轮可以直接验证第一版轻量前线行动闭环的完成态。
- 进入井心室段后的回投口径统一为“返回最近校准的当前锚点”；若晚期复测需要更东侧落点，应允许第二前线锚点直接接管当前回投目标。
- `S10` / `S11` / `S12` / `S13` 载入后，当前回投目标应直接切到井心室前第二锚点，避免为了观察井纺室、井纹架、井系桥或锚场回稳链路先手动重校准一次。
- 若复测只差关键资源或生命 / 防护，优先用调试 HUD 中的 GM 区补状态，不额外生成一次性临时存档。

## 运行时使用方式

当前原型里，按 `Tab` 打开左侧调试面板后，会看到“开发基线”区。

使用步骤：

1. 点击“上一个 / 下一个”切换基线。
2. 点击“载入”把当前运行时重置到该阶段。
3. 如需长期保留，再用现有普通槽位按钮保存到 `slot_01`、`slot_02` 或 `slot_03`。

同一调试侧栏里现在还附带一组开发期 GM 控件：

- 可轮换关键资源并做 `+1 / -1` 数量调整。
- 可一键回满生命和防护。

它只服务本地开发复测，不面向正式玩家。

这套机制默认用于本地开发和人工复测，不面向正式玩家。

## 什么时候用基线

默认规则：

- 只改相位井纺核解析、张力绕轮、纬束残团、井纺梭栓或井纺室断面，优先从 `S10` 开始。
- 只改相位井织核解析、边缕残条、井纹架键栓或井纹架断面，优先从 `S11` 开始。
- 只改相位井结核解析、井系桥结点、系索残股、井系定桩或井系桥断面，优先从 `S12` 开始。
- 只改相位井锚核解析、锚核落尘、井系校锚桩、压力钉或锚场回稳，优先从 `S13` 开始。
- 只改锚场回稳完成态、局部稳定窗口收益、相位井余响片解析或后续入口，优先从 `S14` 开始。
- 只改稳窗读数完成态、锚场回访回充、稳窗现场校准或错序反馈，优先从 `S15` 开始。
- 只改稳窗校准完成态、前线行动台入口或 `S16` 后下一循环提示，优先从 `S16` 开始。
- 只改前线行动回报、稳窗回波探点、回报目标完成、反应器自动推荐、补给反馈或基地-前线-基地最短闭环完成态，优先从 `S16` / `S17` 开始。
- 只改相位井心核解析、分流读数、心棘残片、井心分流栓或井心室断面，优先从 `S9` 开始。
- 只改井芯样本解析、井壁余烬、井底穿钉或井底裂口，优先从 `S8` 开始。
- 只改定位器解析、井涌碎屑、探针整备或内层相位井，优先从 `S7` 开始。
- 只改内层故障轨迹、故障残渣、相位井钥或相位井锁链，优先从 `S6` 开始。
- 只改回传后的深段后半段内容，优先从 `S5`、`S6` 或 `S7` 开始。
- 只改深段入口、相位纤丝、过滤器精炼或覆写栓链，优先从 `S3` 开始。
- 只改外圈回波和更深坐标解析链，优先从 `S2` 开始。
- 只改污染边界、过滤模块或处理点补给链，优先从 `S1` 开始。
- 只改新档、前哨恢复、基础采集、早期 HUD / 地图 / 任务提示，优先从 `S0` 开始。

## 什么时候仍然必须跑空档

以下情况不应只依赖局部基线：

- 改了早期主线顺序或新档初始状态。
- 改了共享任务推进、区域解析、地图目标来源、HUD 全局提示口径。
- 改了存档迁移、存档校验或旧进度补激活逻辑。
- 改动跨越多个阶段，且你怀疑基线之间的衔接本身可能出错。
- 一个较大功能包准备收口，想确认从头到尾仍然通顺。

这类情况仍应补一轮 `S0` 全链路复测，必要时再配合中后段基线做局部追踪。

## 维护规则

新增或调整首小时内容时，如果影响了标准阶段切点，需要同步更新三处：

1. `DevelopmentBaselineCatalog` 的阶段定义和用途描述。
2. `DevelopmentBaselineBuilder` 的任务推进、场景完成态和库存口径。
3. 相关检查与文档，确保基线、任务链和人工复测规则一致。

如果某个阶段已经不再适合作为稳定复测起点，应替换它，而不是继续堆更多一次性临时快照。

## 当前结论

首小时原型后续继续扩展时，人工复测的默认姿势应改为：

**先从最接近问题的开发基线开始，再决定是否需要回到 `S0` 全链路重跑。**
