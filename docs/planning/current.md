## 当前阶段

RadishCatalyst 已于 2026-05-13 完成 `S0` 新档冷启动人工复测，并于 2026-05-16 宣布「资源循环与前线节奏重构」阶段通过。

「基地-前线行动核心循环原型」已通过 `S17` / `S18` / `S19` 证明最小模板可以重复成立。下一阶段定为「基地行动选择原型」：不再横向新增同构短行动，而是把“基地确认”推进成出发前的真实取舍。

```text
当前主线终点 -> 基地选择下一趟行动方案 -> 前线短目标体现选择差异 -> 前线结果反哺基地
```

后续阶段只让 `P0` / `P1` 阻塞推进：崩溃、主线卡死、坏档、任务无法完成或明显资源断档必须立刻修；目标机械、局部拥挤、文案不够顺等 `P2` / `P3` 进入 backlog 或专门打磨周。

## 当前最高优先级

1. 进入「基地行动选择原型」：先做最小可玩的行动前取舍，而不是新系统大框架。
2. 首版已先落地稳场补给 / 相位测绘二选一；下一步只处理 `P0` / `P1` 或围绕这轮选择补自动检查，不扩完整队列。
3. 只让 `P0` / `P1` 问题打断当前阶段；`P2` / `P3` 体验打磨、前线行动台完整队列和更多短行动分支进入 backlog。
4. 保持资源预算、开发基线、HUD / 任务链和存档检查持续通过。

## 当前不做

- 不把联机、官方账号、交易、公会、排行榜、云存档或复杂服务端作为首版前置条件。
- 不把完整 Web 客户端、Android 或官方工具作为 MVP 必须交付项。
- 不把 Godot 客户端核心逻辑押到 C# / .NET。
- 不大规模生产美术资产，不提前批量编写玩家 Wiki 百科条目。
- 不继续沿相位井后段横向堆内容包或为了 `P2` / `P3` 体验项反复打磨。
- 不新增第 4 条同构轻量前线行动，不急着把多个前线行动 console 收束成完整队列系统。
- 不新增电网、管线、自动物流、多设备调度或完整教程弹窗框架，除非它们直接服务下一组基地-前线行动核心循环原型。
- 不把参考仓库或历史旧仓库代码整包迁入本仓库。

## 代码实践约束

正式规范见 `docs/architecture/code-style-and-language-practices.md`。当前阶段尤其需要避免为了拆分 HUD 或验证脚本而制造晦涩封装；新抽象必须能降低复杂度、减少真实重复或隔离清晰边界。

## 当前默认验证

客户端相关改动优先执行聚合验证：

```powershell
pwsh ./scripts/check-client.ps1
```

文档、规划、协作规则或仓库入口改动额外执行：

```powershell
pwsh ./scripts/check-docs.ps1
```

提交前仍需执行仓库文本卫生和 diff 检查：

```powershell
pwsh ./scripts/check-text-files.ps1
git diff --check
```

脚本通过不等于体验完全通过；但个人开发阶段只要求 `P0` / `P1` 得到处理，`P2` / `P3` 体验项不阻塞新循环推进。人工短复测改为阶段闸门或风险触发，不再要求每条轻量行动都手动短跑。
