## 当前阶段

RadishCatalyst 已于 2026-05-13 完成 `S0` 新档冷启动人工复测：人工确认可以从 0 玩到当前主线终点，首小时引导可玩阶段可以收口。

下一阶段进入「资源循环与前线节奏重构」。目标不是继续横向加相位井后段包，而是把已经跑通的线性主线改成更稳、更有变化的基地-远征循环：

```text
稳定资源供给 -> 基地整备形成远征方案 -> 前线目标产生不同玩法和回访收益
```

本阶段优先治理两项体验债：基础零件过度依赖有限晶体簇，以及深段后半段任务同构、往返基地成本过高。

## 当前最高优先级

1. 收口 `S0` 空档测试结论，按阶段通过处理当前修复；下一步开发默认进入「资源循环与前线节奏重构」。
2. 完善基础零件稳定供给：阶段补给批次、缺料提示、第一处富晶残脉、“非早期主线矿点”门槛、相位中继锚点后的污染浆液回收产线、解析类配方减税和后段装配整备包已接入，下一步继续观察装配类消耗和后段误操作缓冲是否足够。
3. 重构深段后段任务结构：裂相脊到稳窗相位序已先改掉复制排布，并已通过 `S5` 到 `S16` 首轮人工复测；后续只在真实试玩中继续观察读数、清障、泄压、压力钉和定序目标是否出现新的机械感。
4. 降低重复往返基地成本：回基地应是整备、选择、升级或批量加工，不是大老远跑回去只按一次制造。
5. 持续维护资源预算检查，新增配方、任务或阶段补给时同步更新预算口径，保证少量误操作也不需要 GM 面板补资源。

## 当前不做

- 不把联机、官方账号、交易、公会、排行榜、云存档或复杂服务端作为首版前置条件。
- 不把完整 Web 客户端、Android 或官方工具作为 MVP 必须交付项。
- 不把 Godot 客户端核心逻辑押到 C# / .NET。
- 不大规模生产美术资产，不提前批量编写玩家 Wiki 百科条目。
- 不继续沿相位井后段横向堆同构内容包。
- 不新增电网、管线、自动物流、多设备调度或完整教程弹窗框架，除非它们直接服务本阶段资源循环与前线节奏目标。
- 不把参考仓库或历史旧仓库代码整包迁入本仓库。

## 代码实践约束

正式规范见 `docs/architecture/code-style-and-language-practices.md`。当前阶段尤其需要避免为了拆分 HUD 或验证脚本而制造晦涩封装；新抽象必须能降低复杂度、减少真实重复或隔离清晰边界。

## 当前默认验证

客户端相关改动优先执行聚合验证：

```powershell
pwsh ./scripts/check-client.ps1
```

文档、规划、协作规则或仓库入口改动额外执行：

```powershell
pwsh ./scripts/check-docs.ps1
```

提交前仍需执行仓库文本卫生和 diff 检查：

```powershell
pwsh ./scripts/check-text-files.ps1
git diff --check
```

脚本通过不等于体验通过；资源余量、重复跑图、配方查找成本和后段目标变化仍需空存档人工试玩判断。

## 下一步建议

- 复核基础零件供给的玩家可见补给方向：富晶残脉已延后到首次晶体采集完成后开放，长期轻量产线已用污染浆液回收接入相位中继锚点后，解析类固定税已移除，后段装配件已并入整备包；下一步观察真实试玩中基础零件缓冲是否仍偏紧。
- 继续让资源预算自动检查跟随配方和任务变化，避免阶段补给、任务件消耗或中间加工新增后重新产生断档。
- 继续复核相位井锁之后的后段节奏：每次回基地应尽量打包成一次远征整备或批量加工，下一趟外勤直接解决现场目标。
- 给深段后段至少做出 4 种目标模板：共振读数、脉冲读数、泄压、路线清障、读数解锁采集、压力清理短守场和现场校准已有第一版，读数 / 清障提示和末段地图可读性已先回修；后续重点转向资源余量观察、阶段复核和必要删改。
- 前线回传锚点继续作为跑图减压工具；新阶段优先让锚点、临时装配或批量加工减少“回基地只做一个零件”的挫败感。
- 每完成一个小阶段，按 `docs/planning/milestone-review-checklist.md` 做兼容、存档、数据和范围复核。

## 阶段退出条件

- 玩家能在无 GM 面板补资源的情况下，从新档完成当前主线，并保留合理基础零件缓冲。
- 基础零件不再由有限晶体簇决定主线能否推进；缺料时 UI、地图或日志能指向明确补给来源。
- 深段后段连续任务不再重复同一套“打怪、采集、回基地制造、返回开锁”结构。
- 回基地行为能形成整备、选择、升级、批量加工或回访收益，而不是单次钥匙制造。
- `check-client.ps1`、`check-docs.ps1`、`check-text-files.ps1`、`git diff --check` 持续通过，阶段边界和协作规则同步更新到 `docs/`、`AGENTS.md` 和 `CLAUDE.md`。
