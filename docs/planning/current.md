## 当前阶段

RadishCatalyst 当前处于「首小时引导可玩」阶段。

第一切片冷启动审计已于 2026-05-05 通过，上一阶段「Godot 第一可玩切片原型骨架实现」退出条件已满足。当前重点不是按分钟数堆内容，而是验证玩家是否能理解并完成：

```text
外出获得资源 -> 回基地加工 -> 远征产物改变下一次外勤结果
```

2026-05-05 空存档试玩显示完整流程扩展后仍不足 10 分钟。该时长仅作为内容密度、理解断点和工程摩擦的观察信号，不再作为阶段硬目标。

- `client/` 已建立 Godot 4.x 原型工程，第一切片主流程、任务链、静态数据、存档运行时、HUD 关键面板和验证脚本已可支撑反复冷启动复测。
- 第一切片已覆盖前哨恢复、晶体采集、基础战斗、基地加工、异常样本分析、基础过滤模块、处理点建设、污染边界挑战、污染残核压制和切片结尾提示。
- 基础反应器已支持任务相关配方自动优先选择，设备面板会标记当前目标推荐配方，降低靠 `R` 反复切换查找配方的摩擦。
- HUD 已开始从原型巨型脚本拆分职责：调试面板、设备面板、反馈面板、右侧状态、小地图、短提示和日志格式化已有独立 presenter 承担文本和展示规则；静态数据检查已覆盖主线任务图可达性、前后置一致性和任务目标来源。
- 当前原型仍是工程演示，不代表正式画面、正式战斗、正式存档 UI 或完整首小时内容。

## 当前最高优先级

1. 保护「基地服务冒险，冒险反哺基地」的核心闭环。
2. 优先处理会阻碍后续内容扩展的工程摩擦：任务链数据维护、正式 HUD / 调试 HUD 边界、验证脚本职责和配方选择体验。
3. 空存档复测时记录阻断、理解断点、无意义等待、重复跑图、配方查找成本和失败补救是否明确；总耗时只作为观察指标。
4. 内容扩展只在闭环可信度不足时补齐，不用等待时间、长文本或纯 UI 抛光硬拉流程。
5. 保持入口文档简约；阶段历史和长清单记录到周志或专题文档。

## 当前不做

- 不把联机、官方账号、交易、公会、排行榜、云存档或复杂服务端作为首版前置条件。
- 不把完整 Web 客户端、Android 或官方工具作为 MVP 必须交付项。
- 不把 Godot 客户端核心逻辑押到 C# / .NET。
- 不大规模生产美术资产，不提前批量编写玩家 Wiki 百科条目。
- 不在首小时同时解锁完整管线、多设备自动化、多天气、多角色 Build、大型任务日志、复杂技能树或联机入口。
- 不主动扩展存档边界校验、电网、管线、自动物流和多设备调度，除非真实改动需要。
- 不把参考仓库或历史旧仓库代码整包迁入本仓库。

## 代码实践约束

正式规范见 `docs/architecture/code-style-and-language-practices.md`。当前阶段尤其需要避免为了拆分 HUD 或验证脚本而制造晦涩封装；新抽象必须能降低复杂度、减少真实重复或隔离清晰边界。

## 当前默认验证

客户端相关改动优先执行聚合验证：

```powershell
pwsh ./scripts/check-client.ps1
```

该脚本按顺序执行：

```powershell
pwsh ./scripts/check-client-data.ps1
pwsh ./scripts/check-client-scenes.ps1
pwsh ./scripts/check-client-save.ps1
pwsh ./scripts/check-client-quests.ps1
pwsh ./scripts/check-client-flow.ps1
pwsh ./scripts/check-godot-client.ps1
```

其中 `check-client-data.ps1` 已覆盖静态数据引用、本地化键、配方解锁来源一致性、任务激活重复声明、区域任务索引、主线任务图一致性和任务目标来源，避免 `unlock_conditions`、`next_quest_ids`、任务 `unlock_effects`、`quest_refs`、主线前后置和目标可获得性脱节。

提交前仍需执行仓库文本卫生和 diff 检查：

```powershell
pwsh ./scripts/check-text-files.ps1
git diff --check
```

脚本通过不等于首小时体验通过；文案理解、节奏感受、配方查找成本和重复跑图仍需空存档人工试玩判断。

## 下一步建议

- 继续收敛验证脚本职责和任务链数据维护；主线任务图和任务目标来源已纳入静态数据检查，后续优先评估内容生产工作流中仍靠人工同步的口径。
- 复测设备面板的当前目标推荐配方标记是否足够清楚；如仍有摩擦，再考虑快捷选择，不先做复杂设备管理。
- 若玩家仍不理解基地加工如何影响下一次外勤，优先补外勤风险、战斗准备价值或资源选择，而不是增加等待时长。
- 每完成一个小阶段，按 `docs/planning/milestone-review-checklist.md` 做兼容、存档、数据和范围复核。

## 阶段退出条件

- 玩家能在无开发者口头解释下完成至少一次「外出获得资源 -> 回基地加工 -> 远征产物改变下一次外勤结果」闭环。
- 玩家能理解基地后勤、设备缺料、地图目标、采集、基础战斗、制造远征产物、污染与防护、处理点建设、危险区域挑战和后续入口。
- 失败或被挡回时，UI、日志、地图标记或设备状态能给出具体补救方向。
- HUD、日志、方向提示、设备面板、小地图、任务完成反馈、撤离反馈和右侧状态摘要之间不遮挡、不信息过载。
- `check-client.ps1`、`check-text-files.ps1`、`git diff --check` 持续通过，阶段边界和协作规则同步更新到 `docs/`、`AGENTS.md` 和 `CLAUDE.md`。
