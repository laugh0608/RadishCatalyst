## 当前不做

- 首版和首个存档 + 联机版本不做 MMO 大世界，但保留更远期大型多人在线的演进口子。
- 不把联机作为首版前置条件。
- 联机方向按 2~5 人协作作为首版设计基准，长期只预留 2~10 人专服或高性能房间扩展。
- 不提前实现官方账号、交易、公会、排行榜或云存档。
- 不引入复杂服务端框架。
- 不把完整游戏 Web 客户端作为 MVP 必须交付项。
- 不把 Godot 客户端核心逻辑押到 C# / .NET。
- 不大规模生产美术资产。
- 不提前批量编写玩家 Wiki 百科条目。
- 不立即实现完整官方工具。
- 不把参考仓库代码整包迁入本仓库。
- 不在首小时同时解锁完整管线、多设备自动化、多天气、多角色 Build、大型任务日志、复杂技能树或联机入口。
- 不优先做完整教程弹窗框架、可回看百科或键位自定义引导，除非首小时计时试玩证明确有阻断。
- 不主动扩展存档边界校验、电网、管线、自动物流和多设备调度。

## 当前默认验证

客户端相关改动优先执行聚合验证：

```powershell
pwsh ./scripts/check-client.ps1
```

该脚本按顺序执行：

```powershell
pwsh ./scripts/check-client-data.ps1
pwsh ./scripts/check-client-scenes.ps1
pwsh ./scripts/check-client-save.ps1
pwsh ./scripts/check-client-quests.ps1
pwsh ./scripts/check-client-flow.ps1
pwsh ./scripts/check-godot-client.ps1
```

其中 `check-client-scenes.ps1` 会校验场景资源引用、Godot 脚本 UID 和原型 HUD 关键面板默认布局，避免状态、地图、设备、任务完成、撤离、补给、提示和日志面板互相遮挡；`check-client-quests.ps1` 用于直接复验任务事件映射、任务目标进度和任务完成规则；`check-client-flow.ps1` 用于复验主线任务链从恢复前哨推进到 `slice_01_complete` 的关键状态，不等同于完整 UI 自动试玩。

提交前仍需执行仓库文本卫生和 diff 检查：

```powershell
pwsh ./scripts/check-text-files.ps1
git diff --check
```

进入「首小时引导可玩」阶段后，自动化脚本仍守住状态链路，但首小时停留时长、文案理解和节奏感受需通过带计时的人工试玩判断；脚本通过不等于首小时通过。

## 阶段退出条件

满足以下条件后，可视为「首小时引导可玩」阶段达成，再决定是否进入更后续的内容生产或长闭环扩展阶段：

- 玩家能在无开发者口头解释下完成首小时闭环，至少完成一次「外出获得资源 -> 基地加工 -> 远征产物改变下一次外勤结果」的可信体验。
- 玩家在首小时内能依次理解：基地后勤、设备缺料、地图目标、采集、基础战斗、回基地加工、制造远征产物、污染与防护、采样与分析、清理 / 平整 / 地基、装备或使用模块 / 药剂、危险区域挑战、后续入口解锁。
- 首小时各段实际停留时长大致落在 `docs/design/onboarding-and-first-hour.md` 给出的目标区间，没有任何一段被严重压缩或拖长到节奏失衡。
- 玩家失败或被挡回时，UI、日志、地图标记或设备状态能给出具体补救方向，没有出现「条件不足」式的含糊提示。
- HUD、日志、方向提示、设备面板、小地图、任务完成反馈、撤离结果反馈和右侧状态短摘要之间没有互相遮挡，也没有形成信息过载。
- 自动化脚本（`check-client.ps1`、`check-text-files.ps1`、`git diff --check`）持续通过，且每次小修都有对应的最小验证记录在周志中。
- 阶段边界、协作规则或验证基线如有调整，已同步更新 `docs/`、`AGENTS.md` 和 `CLAUDE.md`。
