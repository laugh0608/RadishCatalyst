# 2026-W18 Development Log

周期：2026-04-27 至 2026-05-03

## 本周目标

- 将 Godot 客户端从空工程骨架推进到第一可玩切片的可运行原型骨架。
- 建立首批静态数据、数据加载入口和最小状态对象。
- 建立 `Boot` 到 `GameRoot` 的启动链。
- 为客户端静态数据补充可重复验证入口。

## 完成情况

- 新增首批 `client/data/` 静态数据，覆盖物品、流体、配方、建筑、装备、敌人、区域、地图对象、污染、天气和任务。

- 新增 `Boot.tscn`、`GameRoot.tscn`、`boot.gd` 和 `game_root.gd`，将项目主场景接到 `Boot`，并在 `GameRoot` 中初始化数据和最小状态。

- 新增 `Player.tscn`、`VerticalSliceMap.tscn`、`PrototypeInteractable.tscn`、`PrototypeEnemy.tscn` 和 `PrototypeHud.tscn`，形成可移动、可交互的临时切片地图。

- 启用基础过滤模块时会标记污染边界区，为后续污染边界敌人、处理设备或任务推进留出入口。

- 调整加工设备交互提示，单配方设备只提示 `E` 加工，多配方设备才显示 `R` 切换。

## 关键决策

- 第一轮工程实现继续围绕第一可玩切片，不提前扩展完整 MVP 内容量。
- 静态数据先使用 JSON，脚本和后续工具都通过稳定 ID 引用定义。
- 原型地图先使用 `ColorRect` 和标签占位，不提前引入正式美术资产。
- 本机 Godot console 可执行文件路径固定为 `D:\Program Files\Godot\Godot_v4.6.2-stable_win64_console.exe`，用于当前阶段的无头项目加载验证。
- `--headless --path ... --quit` 和 `--check-only --script` 在当前 Windows / Godot 4.6.2 环境会触发引擎层崩溃；当前先采用稳定的 `--import --quit` 作为项目级验证。
- Godot 导入和仓库扫描不适合并行执行；当前验证命令应顺序运行，避免缓存和文件扫描互相争用。
