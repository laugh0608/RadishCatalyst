## 关键决策

- `docs/` 是当前唯一正式文档源。
- `dev` 作为日常开发与文档集成分支。
- `master` / `main` 作为稳定主线，只通过 Pull Request 合并。
- 当前 CI 只设置 `Repo Hygiene`，不引入尚不存在的 Godot 构建或测试门禁。
- 单个源码文件原则上不超过 1000 行，硬上限 1500 行。
- 首版仍以单机完整闭环为目标，联机只做架构预留。
- 联机方向确定为先采用 Minecraft 式多形态联机：首版按 2~5 人协作设计，长期预留 2~10 人专服或高性能房间扩展；更远期大型多人在线只保留演进口子，不进入首版和首个联机版本范围。
- `docs/` 只承载开发者文档，玩家可见内容转入 `wiki/`。
- 面向玩家的官方辅助工具使用 `official-tools/`，不与仓库脚本 `tools/` 混用。
- 游戏数据、玩家 Wiki 和官方工具未来应尽量共享结构化数据源。
- 当前阶段先完成“项目到底是什么、讲什么故事、看起来像什么、如何发布与兼容”的上位定稿，再拆首版 MVP 功能清单。
- 视觉方向确定为清晰克制的工业科幻现场界面，叠加危险异星生态和化工设施可读性。
- 世界观方向确定为前哨工程战斗员在异常星球建立化工基地，通过生产、探索、污染治理和秘境调查逐步揭开催化现象。
- 平台方向确定为 Windows 桌面优先，Web 作为试玩和传播渠道预留，Android 只做远期评估。
- 长期探索方向允许不同星球或大型区域形成差异，但基地设备、UI、图标、数据 ID 和工程识别语言应保持统一；星球差异主要来自生态、地貌、污染类型、资源体系和基地适应需求。
- 核心玩法循环正式收束为“基地准备远征，远征带回样本和资源，基地转化为装备、工艺和通行能力，再推动玩家进入更危险区域”。
- 角色成长方向确定为不做脱离基地的传统刷怪升级，而是通过武器、防护服、模块、消耗品和技能 Build 承接基地后勤。
- 角色与存档方向确定为玩家可拥有多个角色，不同角色可有不同初始能力和天赋；世界进度、基地资源、关键装备和主线推进默认归属世界存档。
- 首版 MVP 范围确定为验证“基地准备 -> 外出探索 -> 采集 / 战斗 / 采样 -> 回基地加工 -> 产出装备 / 药剂 / 模块 -> 解锁更危险区域”的最小闭环，不以完整游戏内容为目标。
- 首版 MVP 必须离线单机可玩并实现本地存档；联机不是首版前置条件，首版和首个存档 + 联机版本都不做 MMO 大世界。
- 首版地形与建设交互采用克制边界：先做可破坏 / 不可破坏、可建造 / 不可建造、简单高低差、清理、平整和基础地板 / 地基，不做复杂地形雕刻、结构承重或多层建筑。
- 参考作品只作为局部体验参照：吸收基地战斗、自动化链路、长期工业目标和现场建设感，但不把项目做成单一参考作品的换皮。
- 时间与天气系统值得保留，但首版先作为轻量环境压力和数据预留处理，不做复杂气候模拟、季节系统或全球天气网络。
- 首版存档边界确定为静态配置使用稳定定义 ID，运行时状态使用世界内实例 ID；世界保存地图、基地、污染、时间天气和任务，角色保存生命、装备、背包和成长，设置独立保存。
- 第一可玩切片确定为“基地平台 -> 晶体矿脉区 -> 采集 / 战斗 / 采样 -> 回基地加工 -> 制造过滤模块 / 药剂 / 地基 -> 进入污染边界 -> 小型精英挑战 -> 解锁下一入口”。
- 首版叙事先服务任务推进和系统解锁，不做完整长篇剧情、多分支叙事或大量 NPC。
- 首小时引导应一次只教一个系统，先让玩家完成可信闭环，再逐步展开化工、污染、地形、天气和长期成长。
- Godot 工程初始化默认使用 GDScript；静态数据放在 `data/`，可保存状态放在 `state/`，规则放在 `systems/`，表现和场景放在 `scenes/` 与 `ui/`，存档逻辑放在 `save/`。
- Windows 桌面版是核心游戏主平台；Web 游戏客户端只做可选试玩 Demo / 技术验证；Android 是远期评估；WebApp 是 Wiki、配方查询器、工艺链规划器和地图 / 资源工具，不是完整游戏客户端。
- .NET / C# 不作为 Godot 客户端首版核心依赖，优先用于外部工具、数据校验、存档迁移、WebApp 后端和后续服务端。
- 每个阶段或里程碑完成后必须做兼容复核，避免前期实现破坏后期平台、存档、数据、任务和工程结构兼容。
- Godot 客户端工程位置固定为 `client/`；Godot 编辑器缓存 `.godot/` 不进入版本库，仓库根 `.editorconfig` 和 `.gitattributes` 继续作为文本规范真相源。
