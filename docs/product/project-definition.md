# Project Definition

更新时间：2026-04-26

## 文档目的

这份文档用于回答 RadishCatalyst 在进入具体 MVP、玩法系统和 Godot 原型之前的上位问题：

- 这到底是一款什么游戏？
- 玩家为什么要玩？
- 项目必须具备哪些元素？
- 它要讲一个什么方向的故事？
- UI 和客户端形态应如何服务这个方向？
- 玩家 Wiki 和官方工具为什么从一开始就要纳入规划？

## 一句话定义

RadishCatalyst 是一款以异星化工基地为核心、以人物探索战斗和角色成长为主线、以生产链规划和未知区域远征为长期驱动的 2D / 2.5D 工业科幻 ARPG。

## 核心体验

玩家应该反复体验到三件事：

1. 外出探索未知区域，面对怪物、环境危险和资源选择。
2. 回到基地，把采集和战斗所得转化为设备、材料、药剂、装备和新工艺。
3. 通过更强的角色和更复杂的生产链，进入更深、更危险、更有价值的区域。

项目不是纯工厂模拟，也不是纯刷怪 ARPG。它的核心是：

**用化工生产链支撑冒险，用冒险压力反过来推动基地升级。**

## 参考定位

本项目可以参考《银河破裂者》《异星工厂》《戴森球计划》《幸福工厂》等作品的局部长处，但不应简单模仿其中任何一款。

正式参考口径以 `reference-positioning.md` 为准。

当前核心判断是：

- 从《银河破裂者》参考基地与角色外出战斗的节奏，但不把首版做成大规模守家敌潮。
- 从《异星工厂》参考自动化链路、物流、能源和污染关系，但不做纯工厂游戏。
- 从《戴森球计划》参考长期工业目标和多星球资源差异，但首版不做星际物流或多星球网络。
- 从《幸福工厂》参考地形、地基和现场建设感，但不采用大型 3D 工厂作为首版目标。
- RadishCatalyst 必须把差异化落在化工链、污染治理、样本分析、角色外勤和基地后勤成长上。

## 必须具备的元素

- 异星环境。
- 化工基地。
- 角色操控、探索与战斗。
- 材料、配方、设备和生产线。
- 未知区域、危险区域和秘境。
- 时间、天气、污染和环境危险。
- 怪物、Boss 和敌对生态。
- 角色成长、装备成长和消耗品成长。
- 可公开沉淀到玩家 Wiki 的知识系统。
- 后续可扩展到官方工具的数据结构。

## 可选但不应首版强求的元素

- 多人协作。
- 专用服务器。
- 官方账号。
- 云存档。
- 玩家交易。
- 大型剧情分支。
- 高复杂度自动化蓝图系统。
- 完整 Mod 平台。

## 故事方向

玩家不是普通幸存者，而是具备工程、战斗和现场处置能力的异星开拓者。

基础故事可以围绕以下问题展开：

- 为什么人类或某个组织必须来到这颗星球？
- 这颗星球为什么有高价值资源？
- 化工基地为什么是生存和探索的前提？
- 怪物和异星生态为什么与资源、污染、催化反应或未知能量有关？
- 玩家逐步深入未知区域时，会发现什么被隐藏的真相？

故事基调应是“工业开拓 + 异星未知 + 危险现场处置”，而不是传统英雄救世或纯沙盒无叙事。

## UI 风格方向

UI 应服务于两种状态：

- 冒险状态：清楚显示生命、能量、装备、技能、快捷物品、危险提示。
- 基地状态：清楚显示设备、管线、输入输出、库存、配方、产能和事故风险。

推荐方向：

- 工程终端感。
- 清晰、克制、高信息密度。
- 适度异星材料和工业符号。
- 不做厚重赛博霓虹，不做纯后台管理系统。
- 不让 UI 抢走 ARPG 操作和地图探索的注意力。

## 客户端与发布形态

当前优先级：

1. Windows 桌面版：核心游戏主平台和首版主目标。
2. Web 试玩 Demo / 技术验证：可选，不作为 MVP 必须交付。
3. Android：远期评估，不作为首版约束。
4. WebApp：主要指 Wiki、配方查询器、工艺链规划器、地图 / 资源工具，不是完整游戏客户端。

原则：

- 游戏必须支持离线单机。
- 本地存档必须稳定。
- 数据 ID、存档结构和配置格式应为未来联机、Wiki 和官方工具预留。
- 首版不依赖官方服务器。
- Steam 是更自然的长期发布目标。
- Web 游戏客户端只适合轻量试玩和传播，不能反向限制 Windows 主体验。
- WebApp 是玩家工具和知识库方向，不等同于完整游戏 Web 端。

## Wiki 与官方工具定位

本项目天然包含大量需要查询和反查的信息：

- 材料。
- 配方。
- 设备。
- 工艺链。
- 区域资源。
- 怪物掉落。
- 秘境和特殊事件。

因此从项目早期就应把玩家 Wiki 和官方工具纳入结构规划。

- `wiki/` 面向玩家知识沉淀。
- `official-tools/` 面向玩家工具规划与未来实现。
- `docs/` 仍然只作为开发者决策、设计、架构和规划文档。

长期目标是：游戏数据、玩家 Wiki 和官方工具共享同一套结构化数据，避免三者内容互相矛盾。

这里的官方工具优先按 WebApp 规划，但它服务查询、规划和反查，不承诺承载完整游戏客户端。
