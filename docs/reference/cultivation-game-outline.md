# RadishGame 游戏开发大纲

## 1. 项目定位

### 1.1 核心目标

- 项目类型：2D / 2.5D、二次元 / ACG 风格、非像素风、修仙题材长期养成游戏
- 核心体验：探索、战斗、成长、养成、经营并行推进
- 初始目标：先做单机或弱联网可玩版本，不直接做完整 MMORPG
- 平台目标：优先支持 Web、Windows、Android

### 1.2 目标产品形态

- 第一形态：单机 / 弱联网修仙 ARPG
- 第二形态：带云存档、聊天、排行榜、活动的在线版本
- 第三形态：逐步增加多人主城、组队副本、宗门、公会、交易等 MMO 要素

### 1.3 核心玩法循环

- 外出探索
- 进入战斗
- 获取资源与掉落
- 修炼、突破、强化角色
- 返回洞府经营与生产
- 再次进入更高风险区域探索

## 2. 技术路线

### 2.1 引擎与语言

- 游戏引擎：Godot 4.x
- 主要开发语言：GDScript
- 不建议首发阶段使用 Godot C#
- 原因：
  - Godot 更适合个人开发者快速进入原型开发
  - 2D 能力强，工程更轻
  - Web、Windows、Android 可统一导出
  - GDScript 学习和迭代成本更低

### 2.2 平台策略

- Web：作为试玩、传播和快速验证平台
- Windows：作为完整体验的主平台之一
- Android：作为后续扩展平台
- Flutter：不作为游戏主引擎，仅在未来需要时用于启动器、账号中心、社区或商店外壳

### 2.3 后端技术建议

- 第一阶段：本地存档，不接服务端
- 第二阶段：增加账号、云存档、排行榜、聊天等弱联网功能
- 推荐后端方案：
  - Go + WebSocket + PostgreSQL + Redis
- 可选方案：
  - Node.js + TypeScript + WebSocket + PostgreSQL + Redis

### 2.4 美术与工具链

- 引擎：Godot 4.x
- 绘制工具：Krita / Photoshop
- 2D 骨骼动画：Spine
- 地图编辑：Tiled 或 Godot TileMap
- 3D 辅助预渲染：Blender
- 文档设计：Markdown + Figma + Excel / CSV
- 版本管理：Git

## 3. 产品范围控制

### 3.1 明确不在首版完成的内容

- 真正意义上的大型 MMORPG 架构
- 开放式自由交易经济体系
- 大规模同屏玩家战斗
- 复杂 PVP 平衡
- 大量地图和海量剧情内容

### 3.2 首版必须保留的内容

- 可玩的战斗
- 明确的修仙成长线
- 至少一个洞府 / 家园玩法
- 至少一个种植或炼丹系统
- 一条可完成的主线
- 基础探索与掉落反馈

### 3.3 MVP 目标

做出一个可以完整跑通 15 到 30 分钟体验的纵向切片，而不是做一堆半成品系统。

## 4. 世界观与题材方向

### 4.1 题材基调

- 修仙世界
- 二次元 / ACG 审美
- 非像素风
- 氛围偏幻想、宗门、秘境、灵兽、灵田、法宝体系

### 4.2 世界结构建议

- 主城 / 宗门区域
- 洞府 / 家园区域
- 野外探索区域
- 副本 / 秘境区域
- 后期扩展多人公共区域

### 4.3 叙事原则

- 先确定主角成长主线
- 用主线推动大世界认知
- 用支线补足人物、门派、地区和情绪节奏
- 不做过多空泛文本堆砌，优先任务、对话、事件驱动叙事

## 5. 核心系统拆分

### 5.1 角色成长系统

- 等级
- 境界
- 修为
- 突破
- 属性成长
- 天赋 / 灵根
- 功法与技能树

### 5.2 战斗系统

- 实时战斗
- 普攻、技能、闪避、受击、击退
- 怪物 AI
- Boss 机制
- 掉落与奖励反馈

### 5.3 探索系统

- 地图推进
- 宝箱、采集点、机关
- 随机事件
- 支线触发
- 秘境探索

### 5.4 任务系统

- 主线任务
- 支线任务
- 日常 / 委托
- 奇遇事件

### 5.5 家园 / 洞府系统

- 洞府升级
- 建筑解锁
- 灵田种植
- 炼丹房
- 灵兽园
- 资源产出循环

### 5.6 种植与生产系统

- 播种、成长、收获
- 材料加工
- 炼丹配方
- 时间推进与成熟判定

### 5.7 灵兽 / 宠物系统

- 获取
- 培养
- 出战
- 派遣
- 辅助生产

### 5.8 装备与法宝系统

- 武器、防具、饰品
- 法宝成长
- 强化、词条、品质

### 5.9 联网扩展系统

- 账号
- 云存档
- 排行榜
- 聊天
- 好友
- 宗门 / 公会
- 组队副本

## 6. 推荐开发阶段

### 6.1 第一阶段：原型验证

目标：验证 Godot 技术路线和核心玩法循环是否成立。

范围：

- 一个主角
- 一张小地图
- 基础移动
- 基础攻击
- 两个技能
- 三种敌人
- 简单掉落
- 一个主线任务
- 一块灵田
- 本地存档

阶段成果：

- 能在 Web 或 Windows 上完成一次完整的探索、战斗、成长、回家园循环

### 6.2 第二阶段：核心玩法闭环

目标：形成修仙题材的中期可玩体验。

范围：

- 境界突破
- 功法学习与升级
- 背包和装备
- 一个 Boss
- 一个洞府建筑升级系统
- 炼丹或种植完整链路
- 更多支线任务

阶段成果：

- 玩家能清晰感受到成长驱动，并愿意重复进行探索与养成循环

### 6.3 第三阶段：弱联网版本

目标：增加在线感，而不是一开始做重 MMO。

范围：

- 账号系统
- 云存档
- 排行榜
- 邮件
- 聊天
- 世界事件广播

阶段成果：

- 游戏从单机成长为弱联网在线产品

### 6.4 第四阶段：多人化扩展

目标：逐步引入 MMORPG-like 特性。

范围：

- 多人主城同屏
- 组队副本
- 宗门系统
- 好友系统
- 交易系统
- 世界 Boss

阶段成果：

- 在不推翻原有单机核心循环的前提下，逐步增加 MMO 深度

## 8. 技术设计原则

### 8.1 开发原则

- 先做最小可玩，不做大而全
- 先做单机闭环，再做联网
- 优先验证玩法，不优先堆内容
- 优先跑通体验，不优先写复杂框架

### 8.2 工程原则

- 数据驱动优先
- 系统模块解耦
- 尽量避免过度抽象
- 战斗、成长、家园、任务分模块实现
- 配置与逻辑分离

### 8.3 平台原则

- 首先保证 Windows 可稳定运行
- 其次保证 Web 可打开试玩
- 最后再处理 Android 适配

## 9. 风险与应对

### 9.1 最大风险

- 目标体量过大
- 美术资源产能不足
- 联网系统复杂度失控
- 内容生产速度远慢于系统开发

### 9.2 应对策略

- 严格控制 MVP
- 不在首版做真 MMORPG
- 前期使用占位资源
- 尽量复用现成工具和资产
- 每个阶段都做可运行版本
