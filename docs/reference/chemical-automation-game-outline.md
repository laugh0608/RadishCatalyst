# RadishGame 化工自动化游戏开发大纲

## 1. 项目定位

### 1.1 核心目标

- 项目类型：2D / 2.5D、科幻工业风、自动化生产与基地经营题材游戏
- 参考方向：《异星工厂》《银河破裂者》
- 差异化方向：引入真正有化工特色的生产、流体、分离、反应、安全与污染系统
- 初始目标：先做单机可玩的自动化基地游戏，不直接做多人联机
- 平台目标：优先 Windows，其次 Web 试玩，Android 作为后续扩展

### 1.2 目标产品形态

- 第一形态：单机异星化工基地自动化游戏
- 第二形态：带基地防御、环境灾害、科技树扩展的完整版本
- 第三形态：后续可选加入协作、排行榜、创意工坊或弱联网功能

### 1.3 核心玩法循环

- 勘探与采集资源
- 建设加工设施
- 完成反应、分离、储运与供能
- 提升产率、纯度与稳定性
- 处理污染、事故与敌对威胁
- 解锁新科技并扩建基地

## 2. 技术路线

### 2.1 引擎与语言

- 游戏引擎：Godot 4.x
- 主要开发语言：GDScript
- 不建议首发阶段引入复杂多人同步
- 原因：
  - 个人开发者更适合先做完整单机系统
  - 该题材的难点在系统设计，不在联机框架
  - Godot 适合做 2D / 2.5D 单机生产模拟与基地经营

### 2.2 平台策略

- Windows：主平台，保证完整体验
- Web：后续导出试玩版，用于验证传播效果
- Android：在操作和 UI 适配成熟后再考虑

### 2.3 后端技术建议

- 第一阶段：纯本地存档
- 第二阶段：如有需要再增加云存档、排行榜、成就同步
- 推荐后端方案：
  - Go + PostgreSQL
- 可选方案：
  - Node.js + TypeScript + PostgreSQL

### 2.4 美术与工具链

- 引擎：Godot 4.x
- 场景与概念绘制：Krita / Photoshop
- 3D 辅助建模：Blender
- UI 设计：Figma
- 数据配置：Excel / CSV / JSON
- 版本管理：Git

## 3. 产品范围控制

### 3.1 明确不在首版完成的内容

- 开放世界超大地图
- 大规模 RTS 集群单位控制
- 多人联机
- 复杂剧情分支
- 过多资源种类和海量公式化系统

### 3.2 首版必须保留的内容

- 资源采集
- 电力系统
- 至少一条固体加工链
- 至少一条流体加工链
- 至少一种污染机制
- 至少一种事故机制
- 至少一种外部威胁
- 可持续扩建的基地循环

### 3.3 MVP 目标

做出一个可以完整运行 20 到 40 分钟的纵向切片，重点验证“化工自动化 + 风险管理 + 基地扩建”循环是否足够有趣。

## 4. 世界观与题材方向

### 4.1 推荐题材

- 异星化工殖民
- 末世工业基地
- 高腐蚀、高毒性、高压环境下的工业扩张

### 4.2 推荐世界结构

- 着陆基地
- 原料采集区
- 加工区
- 储运区
- 发电区
- 防御区
- 危险地貌区

### 4.3 叙事原则

- 用基地扩张推动叙事
- 用科技解锁体现世界认知升级
- 用环境异常、事故与敌袭制造中期目标
- 少做大段文本，优先事件、日志、任务、设备说明驱动叙事

## 5. 核心系统拆分

### 5.1 资源系统

- 固体资源：矿石、晶体、废渣
- 液体资源：卤水、酸液、冷却液、燃料液
- 气体资源：可燃气、毒气、惰性气体
- 中间体与最终产品区分

### 5.2 生产系统

- 采集
- 运输
- 破碎 / 粉碎
- 熔炼 / 预处理
- 化学反应
- 分离与提纯
- 包装 / 储存 / 输送

### 5.3 流体系统

- 管道网络
- 储罐
- 泵
- 流量控制
- 压力变化
- 温度变化
- 液位管理

### 5.4 单元操作系统

- 反应器
- 分离器
- 过滤器
- 离心机
- 精馏塔
- 吸收塔
- 换热器
- 压缩机

### 5.5 能源系统

- 锅炉
- 发电机
- 燃料供应
- 蒸汽或热能循环
- 电网负载平衡

### 5.6 污染系统

- 废气
- 废液
- 固废
- 地表污染扩散
- 环境恶化反馈

### 5.7 事故与安全系统

- 泄漏
- 过压
- 过热
- 爆炸
- 着火
- 连锁事故
- 安全阀
- 紧急停机

### 5.8 基地防御系统

- 环境生物袭击
- 污染引发的敌袭增强
- 防御塔
- 围墙
- 维修系统
- 灭火与隔离系统

### 5.9 科技树系统

- 采集效率
- 自动化等级
- 高级材料
- 安全技术
- 节能技术
- 清洁生产技术

## 6. 差异化设计重点

### 6.1 化工专业特色

- 不是简单配方合成，而是流程链设计
- 不是只有产量，还要考虑纯度、稳定性和副产物
- 不是只有“机器摆放”，还要考虑物料与能量流

### 6.2 游戏化表达原则

- 不把系统做成考试题
- 不要求玩家手算复杂公式
- 用可视化和明确反馈代替专业推导
- 让“优化工艺”成为核心乐趣之一

### 6.3 可游戏化的化工概念

- 温度窗口
- 压力窗口
- 杂质累积
- 副反应
- 催化剂失活
- 换热效率
- 闭路循环
- 废物回收利用

## 7. 推荐开发阶段

### 7.1 第一阶段：原型验证

目标：验证基础自动化循环和流体系统是否成立。

范围：

- 一张小地图
- 三种基础资源
- 采集设施
- 基础输送系统
- 一条固体加工链
- 一条流体加工链
- 简单发电系统
- 本地存档

阶段成果：

- 玩家能从采集到生产出第一种高价值产品

### 7.2 第二阶段：核心玩法闭环

目标：让生产、扩张、风险管理真正形成循环。

范围：

- 更多单元操作设备
- 污染系统
- 事故系统
- 防御系统
- 科技树
- 更多资源中间体

阶段成果：

- 玩家需要在扩产、稳定、安全和防御之间做取舍

### 7.3 第三阶段：内容扩展

目标：增加策略深度和长线驱动力。

范围：

- 更复杂的地图环境
- 高级工艺链
- 稀有资源区域
- 多阶段敌袭
- 高级安全和环保技术

阶段成果：

- 玩家开始进行工艺改造，而不是只靠堆机器扩张

### 7.4 第四阶段：上线打磨

目标：形成完整可发布单机产品。

范围：

- UI 优化
- 教程与新手引导
- 性能优化
- 平衡调整
- 存档稳定性测试
- Web 试玩版本验证

阶段成果：

- 形成一版可发布的完整产品

## 9. 技术设计原则

### 9.1 开发原则

- 先做核心循环，再做内容扩展
- 先做单机，再考虑联网
- 先让系统跑起来，再考虑复杂画面表现
- 先做可理解的自动化，再逐步增加深度

### 9.2 工程原则

- 数据驱动优先
- 配方、设备、资源、污染、事故参数统一配置化
- 物流、流体、电力、安全分模块设计
- 优先可调试、可视化、可扩展的系统结构

### 9.3 平台原则

- 优先 Windows 平台完成开发
- Web 只作为试玩和传播补充
- 移动端在玩法与交互稳定后再评估

## 10. 风险与应对

### 10.1 最大风险

- 玩法过于硬核，劝退普通玩家
- 系统过多，开发周期失控
- 流体与物流系统性能压力过大
- 美术表现不足导致题材门槛更高

### 10.2 应对策略

- 先做清晰易懂的简化版化工逻辑
- 第一版只做少量设备和资源种类
- 用 UI 提示和图形反馈降低理解门槛
- 逐步增加复杂度，不一次堆满所有专业概念
