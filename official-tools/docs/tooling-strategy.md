# Official Tools Strategy

更新时间：2026-04-25

## 目的

RadishCatalyst 涉及大量材料、设备、配方、区域、怪物、秘境和生产链。  
如果只靠游戏内文本，玩家很难理解复杂依赖，也不利于社区分享路线。

官方工具的目标是在游戏外提供可信、可检索、可规划的辅助体验。

## 优先工具

### 1. 在线合成台

核心问题：

- 某个物品如何获得？
- 它有哪些上游材料？
- 它能用于哪些下游配方？
- 是否存在替代路线？
- 需要哪些设备、条件和区域资源？

### 2. 工艺链规划器

核心问题：

- 每分钟产出目标物品 X 个，需要多少设备？
- 哪些材料是瓶颈？
- 会产生哪些副产物？
- 副产物应回收、排放、储存还是转化？
- 哪些设备需要供能、冷却、催化剂或安全条件？

### 3. 地区资源标点

核心问题：

- 某区域有哪些资源？
- 危险等级如何？
- 秘境入口、Boss、特殊事件在哪里？
- 哪些标点适合公开，哪些应保留探索感？

### 4. 怪物与掉落查询

核心问题：

- 某材料由哪些敌人掉落？
- 某区域会出现哪些敌人？
- 敌人弱点、抗性和推荐装备是什么？

## 数据原则

- 游戏、Wiki 和官方工具应共享同一套结构化数据。
- Wiki 页面尽量由结构化数据补全表格，正文再补玩家可读说明。
- 工具不应成为第二套手写数据库。
- 数据应区分 `public`、`spoiler`、`internal` 三种公开等级。
- 后续如支持 Mod，数据需要保留来源字段，例如 `official` / `mod:<id>`。

## 发布形态

优先级：

1. 静态 Web 页面，可随官网或 GitHub Pages 部署。
2. 游戏内嵌 WebView 或本地离线帮助。
3. 后续可选桌面小工具或移动端适配。

首版不需要账号系统，也不依赖服务器。

## UI 风格方向

- 面向玩家，但保持工程感和清晰的信息密度。
- 避免过重的后台管理界面风格。
- 查询、筛选、反查、路径展开是核心交互。
- 工艺链图应清楚表达设备、输入、输出、副产物和循环。
- 地图标点工具应支持图层、筛选和危险等级提示。

## 现阶段不做

- 不做实时联机地图同步。
- 不做玩家账号和云端收藏。
- 不做社区自由编辑 Wiki。
- 不提前承诺完整 Mod 数据编辑器。
- 不在游戏核心玩法未定前实现复杂工具 UI。

## 与游戏开发的耦合点

为了未来工具可行，游戏数据设计时应提前保留：

- 稳定 ID。
- 本地化显示名。
- 分类、标签和公开等级。
- 输入输出材料。
- 设备需求。
- 区域来源。
- 敌人掉落来源。
- 解锁条件。
- 剧透标记。

这些字段不要求现在全部实现，但应在设计文档和数据格式中预留位置。
